# 006 · 동기 check API + 한도 관리 엔드포인트

> commit `72f75d1` — `feat(api): synchronous check API + per-user limit management endpoints`

## 개요

앞선 세 커밋(counter / limit / pattern)을 하나의 결정으로 합치는 커밋. betting-service가
베팅 접수 직전 동기로 부르는 `POST /internal/v1/risk/check`와, 운영자가 한도를 보고 바꾸는
관리 엔드포인트. 이 커밋이 서비스의 **read side(결정 경로)**를 닫는다 — Kafka leg(다음 커밋)만
남는다.

## 작업 순서

1. `RedisUserBetHistory` — 패턴 엔진이 읽는 history의 Redis 구현(read + write).
2. `RiskCheckService` — single-bet-max → stake 윈도우 → selections → 패턴 순서로 결정.
3. DTO(`RiskCheckRequest/Response`, `LimitUpdateRequest`, `UserLimitsResponse`) + 서비스 record.
4. `RiskCheckController` + `LimitController` + `RestExceptionHandler`.
5. `@WebMvcTest`→`standaloneSetup` MockMvc 테스트.

## 작업 내역

### 서비스 입출력 타입 — Command / Outcome / Rejection

```java
public record RiskCheckCommand(
    String userId, String betId, Money stake, List<String> selectionIds, Instant now) {
  public RiskCheckCommand {
    Objects.requireNonNull(userId, "userId");   // … betId/stake/selectionIds/now …
    selectionIds = List.copyOf(selectionIds);   // now를 명시 주입 → fixed clock 테스트
  }
}

public record LimitRejection(
    String reason, Currency currency, long current, long limit, long requested, String action) {}

public record RiskCheckOutcome(
    boolean approved, Optional<LimitRejection> rejection, List<PatternMatch> patternsFlagged) {

  public static RiskCheckOutcome approved(List<PatternMatch> flagged) {
    return new RiskCheckOutcome(true, Optional.empty(), flagged);
  }
  public static RiskCheckOutcome rejectedByLimit(LimitRejection r, List<PatternMatch> matches) {
    return new RiskCheckOutcome(false, Optional.of(r), matches);
  }
  public static RiskCheckOutcome rejectedByPattern(PatternMatch block, List<PatternMatch> matches) {
    LimitRejection synthetic = new LimitRejection(
        "PATTERN_" + block.ruleName().replace('-', '_').toUpperCase(),
        null, 0L, 0L, 0L, PatternAction.BLOCK.name());
    return new RiskCheckOutcome(false, Optional.of(synthetic), matches);
  }
}
```

서비스 입력은 REST/Avro DTO와 분리된 `RiskCheckCommand`다 — 컨트롤러·consumer가 평가 시각을
주입해 서비스를 결정론적으로 유지(fixed clock 테스트). `RiskCheckOutcome`은 세 팩토리로만
만들어 "승인 / 한도 거절 / 패턴 BLOCK 거절"의 세 상태를 명확히 한다. 패턴 BLOCK은 `LimitRejection`
모양을 합성(synthetic)해 응답 매핑을 한 경로로 통일한다.

### RiskCheckService — cheap-to-expensive 결정

```java
// 생성자: (policy, limitResolver, counter, ruleEngine, meters) — publisher는 007에서 추가된다.
public RiskCheckOutcome check(RiskCheckCommand cmd) {
  return checkTimer.record(() -> doCheck(cmd));   // risk_check_latency_seconds Timer로 매 호출 감쌈
}

private RiskCheckOutcome doCheck(RiskCheckCommand cmd) {
  long stake = cmd.stake().amount();
  Currency currency = cmd.stake().currency();

  // 1. single-bet-max: 정책값과 메모리 비교, Redis 왕복 0
  long singleMax = policy.singleBetMax(currency);
  if (stake > singleMax) {
    return reject(new LimitRejection(
        "SINGLE_BET_MAX_EXCEEDED", currency, 0L, singleMax, stake, PatternAction.BLOCK.name()));
  }

  // 2. stake daily/weekly/monthly: 각각 sliding window peek 1회 + resolver 조회
  for (LimitType type : STAKE_LIMITS) {
    String key = LimitKeys.userKey(cmd.userId(), type);
    long current = counter.currentSum(key, type.window(), cmd.now());
    long limit = limitResolver.resolveUser(cmd.userId(), type, currency);
    if (current + stake > limit) {
      return reject(new LimitRejection(
          type.name() + "_LIMIT_EXCEEDED", currency, current, limit, stake, PatternAction.BLOCK.name()));
    }
  }

  // 3. selections-per-minute (count 기반, currency=null)
  int requestedSelections = cmd.selectionIds().size();
  String selKey = LimitKeys.userKey(cmd.userId(), LimitType.SELECTIONS_PER_MINUTE);
  long selCurrent = counter.currentSum(selKey, LimitType.SELECTIONS_PER_MINUTE.window(), cmd.now());
  long selLimit = limitResolver.resolveUser(cmd.userId(), LimitType.SELECTIONS_PER_MINUTE, currency);
  if (selCurrent + requestedSelections > selLimit) {
    return reject(new LimitRejection(
        "SELECTIONS_PER_MINUTE_LIMIT_EXCEEDED", null, selCurrent, selLimit, requestedSelections,
        PatternAction.BLOCK.name()));
  }

  // 4. 패턴 룰: BLOCK이어도 끝까지 fold(SUSPECT/REVIEW도 응답에 첨부), 매 매치 메트릭
  PatternContext ctx =
      new PatternContext(cmd.userId(), cmd.betId(), cmd.stake(), cmd.selectionIds(), cmd.now());
  List<PatternMatch> matches = ruleEngine.evaluate(ctx);
  matches.forEach(m ->
      meters.counter("risk_pattern_flags_total", "rule", m.ruleName(), "action", m.action().name())
          .increment());

  Optional<PatternMatch> block =
      matches.stream().filter(m -> m.action() == PatternAction.BLOCK).findFirst();
  if (block.isPresent()) {
    meters.counter("risk_limit_violations_total",
        "reason", "PATTERN_" + block.get().ruleName().replace('-', '_').toUpperCase()).increment();
    return RiskCheckOutcome.rejectedByPattern(block.get(), matches);
  }
  return RiskCheckOutcome.approved(matches);
}

private RiskCheckOutcome reject(LimitRejection rejection) {
  meters.counter("risk_limit_violations_total", "reason", rejection.reason()).increment();
  return RiskCheckOutcome.rejectedByLimit(rejection, List.of());
}
```

순서가 의도적이다 — 싼 검사를 먼저 해서 비싼 Redis 왕복 전에 거절할 수 있으면 거절한다.
single-bet-max는 정책값과의 메모리 비교라 Redis를 안 탄다. critical path라 왕복 수를
고정·최소로 유지. 패턴 fold는 BLOCK이 있어도 끝까지 돈다(SUSPECT/REVIEW도 응답에 실어야
하니까). BLOCK이 하나라도 있으면 거절, 아니면 승인하되 매치를 flag로 첨부. ADR-0007 관측성:
`risk_check_latency_seconds`(Timer), `risk_limit_violations_total{reason}` /
`risk_pattern_flags_total{rule,action}`.

### RedisUserBetHistory — 패턴 history의 Redis 구현

```java
@Component
public class RedisUserBetHistory implements UserBetHistory, UserBetHistoryWriter {

  private static final String BETS_KEY_PREFIX = "history:user:";
  private static final String BETS_KEY_SUFFIX = ":bets";
  private static final String SELECTION_KEY_INFIX = ":sel:";
  private static final Duration TTL = Duration.ofDays(7);
  private static final char AMOUNT_DELIMITER = '|';

  // history:user:{userId}:bets              score=ms, member="{betId}|{amount}"
  // history:user:{userId}:sel:{selectionId} score=ms, member=betId

  @Override
  public void recordBet(
      String userId, String betId, long stakeAmount, List<String> selectionIds, Instant now) {
    // … null 체크 + stakeAmount>=0 …
    double score = (double) now.toEpochMilli();
    String betsKey = betsKey(userId);
    zset.add(betsKey, encodeBetMember(betId, stakeAmount), score);
    redis.expire(betsKey, TTL);
    for (String selectionId : selectionIds) {
      String selectionKey = selectionKey(userId, selectionId);
      zset.add(selectionKey, betId, score);
      redis.expire(selectionKey, TTL);
    }
  }

  @Override
  public long countBetsBetween(String userId, Instant from, Instant to) {
    Long count = zset.count(betsKey(userId), from.toEpochMilli(), to.toEpochMilli());
    return count == null ? 0L : count;
  }

  @Override
  public List<Long> recentStakeAmounts(String userId, int lookback) {
    if (lookback <= 0) return List.of();
    Set<String> members = zset.reverseRange(betsKey(userId), 0, (long) lookback - 1);  // 최신순
    if (members == null || members.isEmpty()) return List.of();
    List<Long> amounts = new ArrayList<>(members.size());
    for (String member : members) {
      int pipe = member.lastIndexOf(AMOUNT_DELIMITER);
      if (pipe < 0) continue;
      try { amounts.add(Long.parseLong(member.substring(pipe + 1))); }
      catch (NumberFormatException ignored) { /* corruption: skip, don't poison the rule */ }
    }
    Collections.reverse(amounts);   // REV는 newest-first → 계약은 oldest-first
    return List.copyOf(amounts);
  }

  @Override
  public long countSelectionBets(String userId, String selectionId, Duration window, Instant now) {
    long min = now.minus(window).toEpochMilli();
    long max = now.toEpochMilli();
    Long count = zset.count(selectionKey(userId, selectionId), min, max);
    return count == null ? 0L : count;
  }
}
```

user당 sorted set 두 개. `:bets`는 countBetsBetween(ZCOUNT)과 recentStakeAmounts를,
`:sel:{selectionId}`는 countSelectionBets를 받친다. 7일 TTL이 모든 V1 패턴 윈도우(60s~24h)를
덮어 별도 trim 잡이 필요 없다 — ZCOUNT/ZRANGE가 범위 밖 항목을 싸게 건너뛴다.
`recentStakeAmounts`는 `ZRANGE REV`로 최신순을 받아 **역순으로 뒤집어 oldest-first** 계약을
맞춘다. write(`UserBetHistoryWriter.recordBet`)는 다음 커밋의 Kafka consumer가 유일한 호출자.

### REST 계층 — 컨트롤러 + DTO

```java
// RiskCheckRequest: @NotBlank userId/betId, @NotNull Money stake, @NotEmpty List<@NotBlank> selectionIds
// RiskCheckResponse(@JsonInclude NON_NULL): boolean approved, String rejectionReason,
//     LimitInfo limitInfo, List<PatternFlag> patternsFlagged  (+ nested LimitInfo / PatternFlag)
// LimitUpdateRequest: @NotNull LimitType, Currency(nullable), @PositiveOrZero long amount
// UserLimitsResponse(userId, List<Entry>) — Entry(limitType, currency, value, source=OVERRIDE|POLICY)

@RestController @RequestMapping("/internal/v1/risk")
public class RiskCheckController {
  @PostMapping("/check")
  public RiskCheckResponse check(@Valid @RequestBody RiskCheckRequest request) {
    RiskCheckCommand command = new RiskCheckCommand(
        request.userId(), request.betId(), request.stake(), request.selectionIds(), clock.instant());
    return toResponse(service.check(command));   // approved면 reason/limitInfo null
  }
}
```

`LimitController`(`/internal/v1/risk/limits`)는 GET(effective 한도 스냅샷)/PATCH(override
set)/DELETE(clear). **SELECTIONS_PER_MINUTE는 통화 무관**이라 store/clear 시 `Currency.KRW`를
sentinel로 정규화하고(`effectiveCurrency`) GET 응답에선 wire-level currency를 null로 비운다 —
그 컨벤션이 클라이언트 코드로 새지 않게. GET은 `LimitType.values()` × `Currency.values()`를 돌며
override 유무로 `source`(OVERRIDE/POLICY)를 표기.

### RestExceptionHandler — RFC 7807

```java
@RestControllerAdvice
public class RestExceptionHandler {
  @ExceptionHandler(MethodArgumentNotValidException.class)
  public ResponseEntity<ProblemDetail> handleBodyValidation(...) { return problem(VALIDATION_FAILED, detail); }
  @ExceptionHandler(IllegalArgumentException.class) /* → VALIDATION_FAILED */
  @ExceptionHandler(NoHandlerFoundException.class)  /* → 404 NOT_FOUND ProblemDetail */
  @ExceptionHandler(Exception.class)                /* → INTERNAL_ERROR */

  private static ResponseEntity<ProblemDetail> problem(ErrorCode code, String detail) {
    ProblemDetail body = code.toProblemDetail(detail);
    return ResponseEntity.status(code.httpStatus()).contentType(PROBLEM_JSON).body(body);
  }
}
```

모든 미처리 예외를 shared-protocol의 `ProblemDetail`(RFC 7807)로 깔때질한다. Spring 기본
ProblemDetail이 아니라 shared-protocol 것을 쓰는 건 다른 sportsbook 서비스와 **wire 모양을
똑같이** 하기 위해서(ADR-0004). `application/problem+json` content-type 고정.

### 두 가지 Spring MVC 마찰

**(1) Clock 빈 이름 충돌.** `@WebMvcTest`로 컨트롤러를 띄웠더니 application의 `clock` @Bean과
테스트 설정의 clock이 `BeanDefinitionOverrideException`. `MockMvcBuilders.standaloneSetup`으로
바꿔 컨트롤러 인스턴스만으로 MockMvc를 조립 — 컨텍스트 자체를 안 띄우니 충돌이 사라지고 테스트도
빨라졌다.

**(2) @PathVariable 이름 해석.** `-parameters` 컴파일 플래그가 없으면 Spring이 메서드 파라미터
이름을 reflection으로 못 찾아 `GET /limits/{userId}`가 전부 400을 뱉었다. 에러 메시지가
친절했다: `Name for argument of type [String] not specified ... Ensure that the compiler uses
the '-parameters' flag`. pom.xml 한 줄로 해결:

```xml
<properties>
  <maven.compiler.parameters>true</maven.compiler.parameters>
</properties>
```

C/C++엔 없는 JVM 특유의 함정 — 컴파일 시 파라미터 이름을 바이트코드에 보존할지가 플래그로 갈린다.
