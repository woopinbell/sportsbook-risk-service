# 012 · bet-placed 구독 토픽 정렬 (발행측 `.v1` 드리프트 해소)

> commit `1da19e1` — `fix(event): align bet-placed topic with upstream publisher`
> **phase 2**(후속 fix 윈도우, commit-policy phase 모델)의 첫 커밋.

## 개요

risk-service는 `bet.placed`를 구독하도록 설정돼 있었지만, 발행측 betting-service의 실제 토픽은
`BetEventFactory.TOPIC = "bet.placed.v1"`이다(`.v1` family 통일 — gateway 006 참조). 즉 **통합
환경에서 risk는 betting의 실이벤트를 한 건도 수신하지 못한다.** 이 드리프트는 settlement의
`application.yml:80-86` 주석과 `orchestration/notes/spring-kafka.md`가 사실로 박제하고 있었고,
4차 품질 런이 §7(소스 결함 — 보고만)로 올린 것을 phase 2에서 수정한다.

얄궂게도 토픽 매핑 블록의 기존 주석은 *"Keeps the names in one place so consumers and producers
don't drift"* 라고 적혀 있었다 — 이름을 한 곳에 모으는 것만으로는 부족하고, **그 한 곳의 값이
발행측과 묶여 있어야** 드리프트가 안 난다.

## 작업 순서

1. 발행측 ground truth 확인 — `betting-service .../outbox/BetEventFactory.java:27`의
   `TOPIC = "bet.placed.v1"`.
2. `application.yml` `topics.bet-placed` 값을 `bet.placed.v1`로 정렬, 주석을 발행측 상수 참조로 교정.
3. `EventTopics` 컴팩트 생성자의 **폴백 기본값**도 동일 정렬(설정 누락 시에도 옛 토픽으로 돌아가지 않게).
4. 테스트 픽스처(`RiskEventPublisherTest`)의 토픽 리터럴 동기화.
5. `mvn test` 전/후 결과 동일성 확인.

## 작업 내역

### `application.yml`: 구독 토픽 값과 주석

```yaml
# Kafka topic mapping. Keeps the names in one place so consumers and
# producers don't drift. bet-placed must match the upstream publisher:
# betting-service BetEventFactory.TOPIC = "bet.placed.v1".
topics:
  bet-placed: bet.placed.v1
  risk-limit-violated: risk.limit.violated
  risk-pattern-suspected: risk.pattern.suspected
```

소비 지점은 `BetPlacedConsumer`의 `@KafkaListener(topics = "${topics.bet-placed}", ...)` 하나뿐이라
값 한 줄이 곧 구독 계약이다. 주석은 "어디와 같아야 하는가"(발행측 상수)를 명시하도록 바꿨다 —
다음에 누가 이 값을 만질 때 대조할 단일 출처를 가리킨다.

### `EventTopics`: 폴백 기본값

```java
public EventTopics {
  if (betPlaced == null || betPlaced.isBlank()) {
    betPlaced = "bet.placed.v1";
  }
  ...
}
```

`@ConfigurationProperties` record의 컴팩트 생성자 폴백(001 참조)이 옛 토픽을 들고 있으면, 설정이
누락된 환경에서 조용히 드리프트가 재발한다. 값을 고치는 fix는 **폴백까지 함께** 고쳐야 끝난다.

### 테스트 픽스처 동기화

```java
private static final EventTopics TOPICS =
    new EventTopics("bet.placed.v1", "risk.limit.violated", "risk.pattern.suspected");
```

`RiskEventPublisherTest`는 주입된 `EventTopics`로 발행 토픽을 단언하므로 어떤 값이든 자기일관적으로
green이다 — 그래서 이 드리프트를 **단위 테스트가 구조적으로 못 잡았다**(자기 설정을 자기 기준으로
검증). 픽스처를 실계약 값으로 맞춰 코드베이스에서 옛 리터럴을 제거한다.

## 결과

risk의 구독 토픽이 발행측과 일치한다 — 전 스택 기동 시 betting의 `bet.placed.v1` 이벤트가 risk의
sliding window counter·패턴 히스토리에 실제로 반영될 수 있는 전제가 닫혔다. 코드 로직 변경 없음
(값 2곳 + 픽스처 1곳).

## 요약

- 구독 `bet.placed` ↔ 발행 `bet.placed.v1` 드리프트(4차 런 §7 보고)를 yml 값·`EventTopics` 폴백·
  테스트 픽스처 세 지점에서 정렬.
- 주석을 발행측 상수(`BetEventFactory.TOPIC`) 참조로 교정 — "한 곳에 모음"이 아니라 "발행측과
  묶임"이 드리프트 방지의 본질.
- 단위 테스트는 자기 설정 토픽을 자기 기준으로 검증하므로 이 클래스의 결함을 못 잡는다 — 검출은
  전체 스택 e2e의 몫(orchestration 007 "단독 그린 ≠ 통합 그린"의 세 번째 사례).

## 기억·설명 Level

- **L1**: 변경 3지점(yml 값·폴백·픽스처), 소비 지점은 `@KafkaListener(topics = "${topics.bet-placed}")` 하나.
- **L2**: 폴백 기본값까지 고쳐야 fix가 끝나는 이유(설정 누락 환경의 조용한 재발), 테스트 픽스처가
  자기일관적이라 드리프트에 그린인 구조.
- **L3 (외워서)**: **구독·발행 토픽 계약은 왜 단위 테스트가 못 잡고 어디서 잡나** — 양쪽 모두
  자기 설정을 자기 기준으로 검증하므로 단독 그린이고, 불일치는 두 서비스를 같이 띄운 e2e에서만
  드러난다. 방지책은 검출이 아니라 **단일 출처**(발행측 상수/공유 계약에 구독측을 묶기). 면접에서
  "마이크로서비스 간 계약은 어떻게 지키나", "단위 테스트의 한계 사례"로 직결.
