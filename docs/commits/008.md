# 008 · commons-pool2 (Lettuce 커넥션 풀)

> commit `4427fa6` — `build(deps): add commons-pool2 for Lettuce connection pooling`

## 개요

런타임에서만 드러난 의존성 누락을 메우는 build 커밋. `application.yml`이 Lettuce pool을
켜놨는데 `commons-pool2`가 클래스패스에 없어, 부팅 시점에 `NoClassDefFoundError:
GenericObjectPoolConfig`가 났다.

## 작업 순서

1. 부하 테스트용 jar를 띄우다 부팅 실패 — 단위 테스트는 전부 통과하던 상태.
2. 스택 트레이스에서 `GenericObjectPoolConfig` 누락 확인.
3. pom.xml에 `commons-pool2` 추가(버전은 BOM 관리).
4. 재빌드 후 부팅 성공.

## 작업 내역

```yaml
# application.yml — 이미 pool을 켜둔 상태였다
spring.data.redis.lettuce.pool:
  enabled: true
  max-active: 32
  max-idle: 16
  min-idle: 4
```

```xml
<dependency>
  <groupId>org.apache.commons</groupId>
  <artifactId>commons-pool2</artifactId>
</dependency>
```

Spring Data Redis BOM은 `commons-pool2`를 자동으로 끌어오지 않는다 — pool을 쓸 때만 필요한
optional 의존이라 명시해야 한다. 버전은 안 적는다(Spring Boot 3.2 BOM 관리).

**왜 단위 테스트가 이걸 못 잡았나**: pool은 `LettuceConnectionFactory`가 부팅 시점에
초기화한다. 단위 테스트는 `StringRedisTemplate`을 직접 생성하거나 Testcontainers가 단일
연결로 붙으므로 pool 초기화 경로를 안 탄다. **테스트 그린이 런타임 그린을 보장하지 않는다**는
걸 부하 단계에서 확인한 사례. C/C++로 치면 링크는 됐는데 dlopen 시점에 심볼이 없는 상황과 비슷.

행동 변화는 런타임에 국한된다: pool이 붙으면 Lettuce reactor가 동시 sliding-window 연산을
32 max-active / 16 max-idle 풀에 multiplex한다 — 단일 연결에 직렬화하는 대신.

## 결과

jar 부팅 성공. 부하 테스트로 진행 가능.

## 요약

- `application.yml`이 켠 Lettuce pool에 `commons-pool2`가 필요.
- BOM 관리라 버전 명시 안 함.
- 단위 테스트는 pool 경로를 안 타서 못 잡음 — 런타임에서 발견.
