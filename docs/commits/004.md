# 004 · per-subject 한도 override + 정책 fallback resolver

> commit `50505b7` — `feat(limit): Redis-backed per-subject overrides + policy-fallback resolver`

## 개요

counter(003)가 "지금 얼마 썼나"를 준다면, 이 커밋은 "한도가 얼마인가"를 준다. 운영자(admin-api)가
특정 user/market에 거는 override를 Redis hash에 저장하고, `LimitResolver`가 **override →
yaml 기본값** 순으로 effective limit을 합성한다.

## 작업 순서

1. `LimitOverrideStore` 인터페이스 — 저장 추상화(테스트 더블 교체 가능).
2. `RedisLimitOverrideStore` — Redis hash 구현.
3. `LimitResolver` — override 위에 정책 fallback 합성.
4. Mockito 단위 테스트(resolver) + Testcontainers 테스트(store).

## 작업 내역

### LimitOverrideStore — 저장 추상화

```java
public interface LimitOverrideStore {
  Optional<Long> findUserOverride(String userId, LimitType type, Currency currency);
  Optional<Long> findMarketOverride(String marketId, LimitType type, Currency currency);
  void setUserOverride(String userId, LimitType type, Currency currency, long amount);
  void setMarketOverride(String marketId, LimitType type, Currency currency, long amount);
  void clearUserOverride(String userId, LimitType type, Currency currency);
  void clearMarketOverride(String marketId, LimitType type, Currency currency);
}
```

find는 `Optional`(미설정 = empty), set/clear는 mutation. 인터페이스를 저장-불가지론적으로 둬서
`LimitResolver`가 Redis 구현 위든 인메모리 테스트 더블 위든 같은 방식으로 fallback을 합성한다.
admin-api만 이 store를 mutate하고 나머지는 read-only.

### RedisLimitOverrideStore — subject당 hash 하나

```
limit:override:user:{userId}      Redis hash
    field = "{LIMIT_TYPE}:{CURRENCY}"   예: "STAKE_DAILY:KRW"
    value = long minor unit을 십진 문자열로
limit:override:market:{marketId}  같은 모양, 분리된 prefix
```

```java
@Component
public class RedisLimitOverrideStore implements LimitOverrideStore {

  private static final String USER_KEY_PREFIX = "limit:override:user:";
  private static final String MARKET_KEY_PREFIX = "limit:override:market:";

  private final HashOperations<String, String, String> hashOps;

  public RedisLimitOverrideStore(StringRedisTemplate redis) {
    this.hashOps = redis.opsForHash();
  }

  // find* → read(key, field) / set* → write(key, field, amount) / clear* → hashOps.delete(key, field)
  // (user/market 6개 메서드는 prefix만 다른 위임)

  private Optional<Long> read(String key, String field) {
    String raw = hashOps.get(key, field);
    if (raw == null || raw.isBlank()) {
      return Optional.empty();
    }
    try {
      return Optional.of(Long.parseLong(raw));
    } catch (NumberFormatException e) {
      throw new IllegalStateException(
          "Limit override at " + key + "[" + field + "] is not a long: '" + raw + "'", e);
    }
  }

  private void write(String key, String field, long amount) {
    if (amount < 0) {
      throw new IllegalArgumentException(
          "Limit override must be non-negative, got " + amount + " for " + key + "[" + field + "]");
    }
    hashOps.put(key, field, Long.toString(amount));
  }

  private static String field(LimitType type, Currency currency) {
    return type.name() + ":" + currency.name();
  }
}
```

subject 하나의 모든 override가 hash 하나에 모인다. 그래서 `HGET`은 단일 override를 O(1)로
주고(critical path 적합), 제재된 user의 override 전체를 `DEL` 한 번으로 지울 수 있다 — 런타임
counter key는 안 건드리고. field에 LimitType과 Currency를 같이 인코딩한 게 이걸 가능하게 한다.

값을 십진 문자열로 저장하는 건 sliding window의 member 인코딩과 같은 컨벤션이라 파싱 헬퍼가
일관된다. 비-숫자 값이 들어오면(운영 툴 버그) `read`가 `IllegalStateException`을 던져 **조용히
틀린 한도를 반환하지 않는다**(fail loud). `write`는 음수를 거부한다.

### LimitResolver — fallback 합성

```java
@Component
public class LimitResolver {

  private final LimitOverrideStore overrides;
  private final RiskLimitProperties policy;

  public LimitResolver(LimitOverrideStore overrides, RiskLimitProperties policy) {
    this.overrides = overrides;
    this.policy = policy;
  }

  public long resolveUser(String userId, LimitType type, Currency currency) {
    return overrides
        .findUserOverride(userId, type, currency)
        .orElseGet(() -> policyDefault(type, currency));
  }

  public Optional<Long> resolveMarket(String marketId, LimitType type, Currency currency) {
    return overrides.findMarketOverride(marketId, type, currency);  // 기본값 없음
  }

  private long policyDefault(LimitType type, Currency currency) {
    return switch (type) {
      case STAKE_DAILY -> policy.stakeDaily(currency);
      case STAKE_WEEKLY -> policy.stakeWeekly(currency);
      case STAKE_MONTHLY -> policy.stakeMonthly(currency);
      case SELECTIONS_PER_MINUTE -> policy.selectionsPerMinute();
    };
  }
}
```

user는 override → yaml 기본값. **market은 글로벌 기본값이 없다** — 정책은 user 한도만 다루고,
마켓별 상한은 운영자가 명시적으로 거는 opt-in이다. 그래서 `resolveMarket`은 `Optional<Long>`을
반환해 "이 마켓엔 상한 없음"을 무한대 합성 없이 싸게 표현한다.

`policyDefault`는 `LimitType`에 대한 **exhaustive switch**(Java 17)라 새 sliding window 한도를
추가하면 fallback을 안 적은 채로는 **컴파일이 안 된다**. C/C++의 enum switch에서 default
빠뜨려도 경고만 나는 것과 달리, 명시적 dispatch를 강제한다. SELECTIONS_PER_MINUTE는 통화를
무시하고 정수 정책값을 그대로 쓴다.

> 한계(retrospective와 연결): `resolveMarket`은 만들고 테스트까지 했지만, 다음 커밋의
> `RiskCheckService`가 결국 user 한도만 검사한다. market 경로는 저장 계층까지만 연결되고
> 결정 경로엔 미연결로 남았다. 만들다 만 인터페이스다.

## 요약

- subject당 hash 하나 — O(1) 조회, `DEL` 한 번으로 전체 해제.
- user: override → yaml fallback. market: override only(Optional).
- exhaustive switch로 새 한도 추가 시 컴파일 강제.
- 비-숫자 저장값은 fail loud.
- (한계) market 경로는 저장까지만, check 미연결.
