### docker-compose — Redis + KRaft Kafka (두 listener)

```yaml
services:
  redis:
    image: redis:7-alpine
    ports: ["6390:6379"]          # 기본 6379에서 bump — dev 컨테이너와 충돌 방지
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
  kafka:
    image: confluentinc/cp-kafka:7.6.1
    container_name: risk-load-kafka
    ports: ["9094:9092"]
    environment:
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_LISTENERS: INTERNAL://0.0.0.0:9092,HOST://0.0.0.0:9094,CONTROLLER://0.0.0.0:9093
      KAFKA_ADVERTISED_LISTENERS: INTERNAL://risk-load-kafka:9092,HOST://localhost:9094
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,INTERNAL:PLAINTEXT,HOST:PLAINTEXT
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_CONTROLLER_QUORUM_VOTERS: 1@localhost:9093
      KAFKA_INTER_BROKER_LISTENER_NAME: INTERNAL
      CLUSTER_ID: "MkU3OEVBNTcwNTJENDM2Qk"      # 유효한 base64 22자 UUID
```

**KRaft 부팅의 두 함정.** (1) cluster id가 유효한 UUID가 아니어서 broker가 부팅조차 안 됐다
(`Cluster ID string ... does not appear to be a valid UUID`). base64 인코딩된 22자 UUID로 교체.
(2) advertised listener의 localhost 함정 — 처음엔 `localhost:9094` 하나만 뒀다. 그런데 컨테이너
*내부*에서 도는 perf-test의 admin client가 broker metadata를 받아 `localhost:9094`로 다시
dial하려다 무한 대기했다(컨테이너 안의 localhost는 broker 자신이 아니다). INTERNAL(컨테이너
내부 exec용)/HOST(host의 risk-service jar용) 두 listener로 분리. 도커 네트워킹의 반복 교훈 —
"localhost"는 호출하는 쪽이 컨테이너 안인지 밖인지에 따라 완전히 다른 대상이다.

### check_latency.js — realistic vs saturation 2단

```javascript
export const options = {
  scenarios: {
    realistic_1000rps: {
      executor: 'constant-arrival-rate',
      rate: 1000, timeUnit: '1s', duration: '60s',
      preAllocatedVUs: 200, maxVUs: 500, gracefulStop: '5s',
    },
    saturation_5000vu: {
      executor: 'constant-vus',
      vus: 5000, duration: '30s', startTime: '70s', gracefulStop: '5s',
    },
  },
  thresholds: {
    'http_req_duration{scenario:realistic_1000rps,expected_response:true}': ['p(99)<30', 'p(95)<15', 'p(50)<5'],
    'http_req_failed{scenario:realistic_1000rps}': ['rate<0.001'],
    'http_req_failed{scenario:saturation_5000vu}': ['rate<0.01'],   // saturation은 SLO 없음
    checks: ['rate>0.999'],
  },
  summaryTrendStats: ['min', 'avg', 'med', 'p(90)', 'p(95)', 'p(99)', 'max'],
};

export default function () {
  // userId 공간을 흩어 sliding-window 키가 한 sorted set에 몰리지 않게 — hot key는 측정하려는
  // realistic 동시성을 깨뜨린다.
  const userId = `u-${__VU}-${__ITER % 1000}`;
  const body = JSON.stringify({
    userId, betId: `b-${__VU}-${__ITER}`,
    stake: { amount: 10000, currency: 'KRW' },
    selectionIds: [SELECTION_POOL[__ITER % SELECTION_POOL.length]],
  });
  const res = http.post(`${BASE}/internal/v1/risk/check`, body, { headers: { 'Content-Type': 'application/json' } });
  check(res, { 'status is 200': (r) => r.status === 200, 'approved field present': (r) => r.json('approved') !== undefined });
}
```

처음엔 5000 VU constant로만 돌렸더니 p99가 2.37s로 나왔다 — 5000 VU를 동시에 때리면 그건
realistic load가 아니라 queue saturation이다. 두 시나리오로 분리: **realistic_1000rps**는
`constant-arrival-rate`라 k6가 VU 수를 도착률에 맞춰 조절 → latency가 클라이언트 큐잉이 아니라
**서비스 작업**을 반영(README가 인용하는 p99 < 30ms 목표). **saturation_5000vu**는 constant
5000 VU, SLO 없이 queue cliff만 보여줘 capacity planning 자료로. userId fanning이 핵심 —
hot key 하나면 측정이 무의미해진다.

### consumer_throughput.sh — perf-test 래퍼

```bash
set -euo pipefail
RECORDS=${RECORDS:-10000}
RATE=${RATE:--1}
PAYLOAD_SIZE=${PAYLOAD_SIZE:-256}
TOPIC=${TOPIC:-bet.placed}
INTERNAL_BOOTSTRAP=${INTERNAL_BOOTSTRAP:-risk-load-kafka:9092}   # 컨테이너 내부 exec → INTERNAL listener

docker exec risk-load-kafka kafka-producer-perf-test \
  --topic "${TOPIC}" --num-records "${RECORDS}" --record-size "${PAYLOAD_SIZE}" \
  --throughput "${RATE}" \
  --producer-props bootstrap.servers="${INTERNAL_BOOTSTRAP}" acks=all linger.ms=5 batch.size=65536
```

k6엔 first-party Kafka producer가 없어 cp-kafka 컨테이너 안의 `kafka-producer-perf-test`를
쓴다. perf-test가 produce하는 건 합성 바이트(Avro 아님)다 — risk-service consumer는 그걸 Avro
역직렬화하다 실패한다(**의도된** 동작). 이 시나리오는 producer/broker throughput만 측정한다.
"10만 events를 실제로 다 consume"하는 증명은 아니고, Avro 호환 producer는 V2 후보로
`load-test/README.md`에 박제했다.

Level 3 race 정합성(100 동시 record → 정확한 합)은 003 커밋의
`SlidingWindowCounterTest#hundredConcurrentRecordsLandExactSum`이 이미 증명. "한도 임박 동시
베팅 모두 거절"은 별도 harness가 필요해 후속으로 미뤘다(BEST.md에 박제). 결과 raw는
`load-test/results/2026-05-29/`, 요약·단일 출처는 `BEST.md`.
