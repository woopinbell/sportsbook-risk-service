# 007 · Kafka event sourcing + 위험 이벤트 publish

> commit `30cbcea` — `feat(events): Kafka event sourcing for counters + risk event publishing`

## 개요

서비스의 **write side**. betting-service가 발행하는 `bet.placed`를 consume해 sliding window
counter와 패턴 history를 채우고(event sourcing), check에서 나온 위반/패턴을
`risk.limit.violated` / `risk.pattern.suspected`로 publish한다. 이걸로 push 모델(HTTP pull이
아니라 Kafka)이 완성된다 — betting-service에 부담을 안 주고 자체 집계를 유지.

## 작업 순서

1. `AvroCodec` — SpecificRecord ↔ binary 단일 코덱.
2. `EventTopics` — 토픽 이름 `@ConfigurationProperties`.
3. `BetPlacedConsumer` — `@KafkaListener`, counter/history 갱신.
4. `RiskEventPublisher` — 내부 모델 → wire 스키마 번역 후 publish.
5. `RiskCheckService`에 publisher 주입.
6. Mockito 단위 테스트 + Avro 왕복 테스트.

## 작업 내역

### AvroCodec — 왜 plain bytes인가

```java
final class AvroCodec {

  private AvroCodec() {}

  static <T extends SpecificRecordBase> byte[] encode(T record) {
    try (ByteArrayOutputStream out = new ByteArrayOutputStream()) {
      BinaryEncoder encoder = EncoderFactory.get().binaryEncoder(out, null);
      SpecificDatumWriter<T> writer = new SpecificDatumWriter<>(record.getSchema());
      writer.write(record, encoder);
      encoder.flush();
      return out.toByteArray();
    } catch (Exception e) {
      throw new IllegalStateException(
          "Failed to encode " + record.getClass().getSimpleName() + " to Avro", e);
    }
  }

  static <T extends SpecificRecordBase> T decode(byte[] payload, Class<T> type) {
    try {
      T template = type.getDeclaredConstructor().newInstance();
      SpecificDatumReader<T> reader = new SpecificDatumReader<>(type);
      reader.setSchema(template.getSchema());
      BinaryDecoder decoder = DecoderFactory.get().binaryDecoder(payload, null);
      return reader.read(null, decoder);
    } catch (Exception e) {
      throw new IllegalStateException("Failed to decode " + type.getSimpleName() + " from Avro", e);
    }
  }
}
```

V1은 single-schema-per-topic이고 Schema Registry가 없다(ADR-0014). 모든 producer/consumer가
같은 shared-protocol 버전을 링크하므로 스키마 불일치 자체가 발생할 수 없다 → schema id prefix
없이 바이트만 보낸다. decode는 target 타입에 바인딩한 `SpecificDatumReader`로 generic 추론을
좁게 유지한다. 모든 직렬화를 이 한 클래스에 모은 게 V2 Apicurio 전환 때 값을 한다(두 메서드만
교체하면 consumer/publisher는 거의 불변 — change-cost.md 시나리오 3).

### BetPlacedConsumer — event sourcing

```java
@Component
public class BetPlacedConsumer {

  private static final List<LimitType> STAKE_LIMITS =
      List.of(LimitType.STAKE_DAILY, LimitType.STAKE_WEEKLY, LimitType.STAKE_MONTHLY);

  private final SlidingWindowCounter counter;
  private final UserBetHistoryWriter history;

  @KafkaListener(
      topics = "${topics.bet-placed}",
      groupId = "${spring.kafka.consumer.group-id}",
      containerFactory = "kafkaListenerContainerFactory")
  public void onBetPlaced(@Payload byte[] payload, @Header(KafkaHeaders.RECEIVED_KEY) String key) {
    BetPlacedRequested event = AvroCodec.decode(payload, BetPlacedRequested.class);
    String userId = event.getUserId().toString();
    String betId = event.getBetId().toString();
    long stakeAmount = event.getStake().getAmount();
    Instant now = event.getRequestedAt();
    List<String> selectionIds = event.getSelections().stream().map(this::selectionId).toList();

    for (LimitType type : STAKE_LIMITS) {
      counter.record(
          LimitKeys.userKey(userId, type),
          LimitKeys.encodeMember(betId, stakeAmount),
          stakeAmount, type.window(), now);
    }
    int selectionCount = selectionIds.size();
    if (selectionCount > 0) {                       // 빈 슬립은 selection counter skip
      counter.record(
          LimitKeys.userKey(userId, LimitType.SELECTIONS_PER_MINUTE),
          LimitKeys.encodeMember(betId, selectionCount),
          selectionCount, LimitType.SELECTIONS_PER_MINUTE.window(), now);
    }
    history.recordBet(userId, betId, stakeAmount, selectionIds, now);
  }
}
```

partition key = userId(ADR-0006 — 한 user의 스트림 순서 보장). manual ack라 Redis 쓰기가
성공한 뒤에야 offset을 커밋 → 일시적 Redis 장애 시 offset이 안 넘어가 broker가 재전달.

**selection이 있을 때만** selections counter를 record하는 게 미묘한 결정이다. 빈 selection
리스트(예외적 Single)로 amount=0을 ZADD하면, Lua 스크립트의 만료 DECRBY가 0짜리 멤버를
되돌릴 수 없어 sum 불변식이 어긋난다. 그래서 0 record 자체를 막는다(003의 불변식 보호).

### RiskEventPublisher — 내부 모델 ↔ wire 스키마의 두 번역

```java
public void publishLimitViolated(String userId, LimitRejection rejection, Instant occurredAt) {
  Optional<RiskLimitType> wireType = mapLimitType(rejection.reason());
  if (wireType.isEmpty()) {                          // wire에 없는 타입은 조용히 드롭(로그만)
    log.debug("Skipping risk.limit.violated for '{}' (not in shared-protocol RiskLimitType)", rejection.reason());
    return;
  }
  Money requestedAmount = Money.newBuilder()
      .setAmount(rejection.requested())
      .setCurrency(rejection.currency() == null ? "" : rejection.currency().name())
      .build();
  RiskLimitViolated event = RiskLimitViolated.newBuilder()
      .setUserId(userId).setLimitType(wireType.get())
      .setCurrentValue(rejection.current()).setLimitValue(rejection.limit())
      .setRequestedAmount(requestedAmount).setOccurredAt(occurredAt).build();
  kafka.send(topics.riskLimitViolated(), userId, AvroCodec.encode(event));
}

public void publishPatternSuspected(String userId, PatternMatch match, Instant occurredAt) {
  Optional<RiskPatternType> wireType = mapPatternRule(match.ruleName());
  if (wireType.isEmpty()) {                          // 모르는 룰은 WARN 후 skip(조용히 드롭 안 함)
    log.warn("Skipping risk.pattern.suspected for unknown rule '{}'", match.ruleName());
    return;
  }
  Map<String, String> evidence = new HashMap<>();
  evidence.put("reason", match.reason());
  evidence.put("action", match.action().name());
  RiskPatternSuspected event = RiskPatternSuspected.newBuilder()
      .setUserId(userId).setPatternType(wireType.get())
      .setEvidence(evidence).setOccurredAt(occurredAt).build();
  kafka.send(topics.riskPatternSuspected(), userId, AvroCodec.encode(event));
}

static Optional<RiskLimitType> mapLimitType(String internalReason) {
  if (internalReason == null) return Optional.empty();
  if (internalReason.startsWith("STAKE_DAILY")) return Optional.of(RiskLimitType.STAKE_DAILY);
  if (internalReason.startsWith("SELECTIONS_PER_MINUTE")) return Optional.of(RiskLimitType.SELECTIONS_PER_MINUTE);
  return Optional.empty();                           // STAKE_WEEKLY/MONTHLY/SINGLE_BET_MAX → 드롭
}

static Optional<RiskPatternType> mapPatternRule(String ruleName) {
  if (ruleName == null) return Optional.empty();
  return switch (ruleName) {
    case "rapid-betting" -> Optional.of(RiskPatternType.RAPID_BETTING);
    case "sudden-stake-increase" -> Optional.of(RiskPatternType.SUDDEN_STAKE_INCREASE);
    case "repeated-same-selection" -> Optional.of(RiskPatternType.REPEATED_SAME_SELECTION);
    default -> Optional.empty();
  };
}
```

Avro `RiskLimitType`이 STAKE_WEEKLY/MONTHLY/SINGLE_BET_MAX를 모른다. 그 위반들은 로그·메트릭엔
남지만 **publish하지 않는다** — 공개 카탈로그(wire)는 shared-protocol을 단일 출처로 두고 V2
evolution으로 넓히는 게 맞다고 판단(ADR-0014). 매핑은 prefix 기반(reason "STAKE_DAILY_..." →
STAKE_DAILY). 패턴 룰 이름(yaml 키)은 upper-snake `RiskPatternType`으로 switch 매핑, 모르는
룰은 WARN 후 skip(조용히 드롭 안 함). reason/action은 .avsc의 스키마-less `evidence` Map에
실어 — 새 룰이 evidence 키를 추가해도 Avro 스키마 변경이 필요 없다.

### Avro generated 타입이 가정과 달랐다 (재작업)

shared-protocol의 Avro plugin이 `timestamp-millis`를 `long`이 아니라 `java.time.Instant`로
매핑(jsr310)하도록 설정돼 있었다. `setOccurredAt(long)` / `getRequestedAt()`가 long을 준다고
가정하고 짰다가 컴파일 에러 4건. sources jar를 풀어 generated setter 시그니처를 직접 확인하고
`setOccurredAt(Instant)` / `event.getRequestedAt()`(Instant 반환)으로 고쳤다. 교훈: 의존
라이브러리의 generated 코드는 가정하지 말고 먼저 열어볼 것.

### RiskCheckService 통합 (006 대비 diff)

```java
// 생성자에 RiskEventPublisher publisher 파라미터 추가, 필드 보관.
// reject가 cmd를 받아 publish하도록 변경:
private RiskCheckOutcome reject(RiskCheckCommand cmd, LimitRejection rejection) {
  meters.counter("risk_limit_violations_total", "reason", rejection.reason()).increment();
  publisher.publishLimitViolated(cmd.userId(), rejection, cmd.now());   // ← 추가
  return RiskCheckOutcome.rejectedByLimit(rejection, List.of());
}
// 패턴 fold 루프에서 매 매치마다 publish 추가:
for (PatternMatch m : matches) {
  meters.counter("risk_pattern_flags_total", "rule", m.ruleName(), "action", m.action().name()).increment();
  publisher.publishPatternSuspected(cmd.userId(), m, cmd.now());        // ← 추가 (BLOCK뿐 아니라 SUSPECT/REVIEW도)
}
```

publisher를 주입받아 매 reject 지점에서 `risk.limit.violated`를(매핑 안 되는 타입은 publisher가
조용히 드롭), 엔진이 반환한 매 PatternMatch마다 `risk.pattern.suspected`를 publish — BLOCK뿐
아니라 SUSPECT/REVIEW도. downstream 알림이 정작 원하는 건 그 둘이니까. (각 reject 호출에 cmd가
전달되도록 호출부도 `reject(cmd, ...)`로 바뀐다.)

## 기억·설명 Level

- **L1**: consume 1개(bet.placed), publish 2개. partition key=userId.
- **L2**: event sourcing(push) vs HTTP pull 선택, manual ack 재전달, 빈 selection skip 이유.
- **L3 (외워서)**: **wire 스키마 ↔ 내부 모델 불일치를 의도적으로 둔 이유**(shared-protocol 단일
  출처 + V2 evolution), schema-less evidence Map의 진화 친화성, plain Avro(no registry) V1 근거.
  면접 단골: "분산 트랜잭션/이벤트 소싱"·"스키마 진화 전략".
