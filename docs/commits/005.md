## 작업 내역

### 계약 — PatternContext / PatternMatch / PatternRule

```java
public record PatternContext(
    String userId, String betId, Money stake, List<String> selectionIds, Instant now) {
  public PatternContext {
    selectionIds = List.copyOf(selectionIds);   // 불변 방어 — 룰은 ctx를 mutate 못 함
  }
}

public record PatternMatch(String ruleName, PatternAction action, String reason) {}

public interface PatternRule {
  Optional<PatternMatch> evaluate(PatternContext context);   // 미발동 = empty
}
```

`PatternContext`는 룰이 베팅 하나를 평가할 때 보는 전부를 담은 flat·immutable 스냅샷이다.
`now`를 명시 파라미터로 받아 단위 테스트가 `Clock` 배선 없이 fixed instant를 넘긴다. `stake`는
`Money`지만 룰은 `amount()`(long minor unit)만 읽고 통화는 무시한다(통화 스코프는 정책 계층).
`PatternMatch.reason`은 check 응답과 `risk.pattern.suspected` 이벤트 양쪽에 쓰이므로 통화
포매팅·locale을 안 넣는다. `ruleName`은 yaml 키와 동일해 대시보드가 source로 group 가능.

### RuleEngine — 전략 패턴 + Spring 자동 수집

```java
@Component
public class RuleEngine {

  private final List<PatternRule> rules;

  public RuleEngine(List<PatternRule> rules) {   // Spring이 모든 PatternRule 빈을 주입
    this.rules = List.copyOf(Objects.requireNonNull(rules, "rules"));
  }

  public List<PatternMatch> evaluate(PatternContext context) {
    Objects.requireNonNull(context, "context");
    return rules.stream()
        .map(rule -> rule.evaluate(context))
        .filter(Optional::isPresent)
        .map(Optional::get)
        .toList();
  }
}
```

Spring이 `List<PatternRule>`을 주입하면 컨테이너에 등록된 모든 `PatternRule` 빈이 자동으로
들어온다. 새 룰을 추가하려면 `@Component` 클래스 하나만 만들면 되고 엔진은 안 건드린다 —
C++의 plugin 등록을 수동 배열에 push하는 대신 DI 컨테이너가 해주는 셈.

**엔진은 BLOCK에도 short-circuit하지 않고 모든 매치를 반환한다.** 이유: BLOCK의 의미가
호출자마다 다르다. check API에서 BLOCK은 베팅 거절(critical path)이지만, `bet.placed` consumer
입장에선 베팅이 이미 접수된 뒤라 BLOCK은 그저 publish할 이벤트의 심각도(severity)만 바꾼다.
엔진을 순수 fold로 두면 양쪽 호출자의 정책을 엔진에 새지 않게 하고, `List<PatternRule>`만으로
단위 테스트할 수 있다.

### UserBetHistory — 좁은 read 인터페이스

```java
public interface UserBetHistory {
  long countBetsBetween(String userId, Instant from, Instant to);                    // rapid-betting
  List<Long> recentStakeAmounts(String userId, int lookback);                        // sudden-stake (oldest-first)
  long countSelectionBets(String userId, String selectionId, Duration window, Instant now); // repeated
}
```

룰이 Redis나 counter를 직접 만지지 않고 이 인터페이스만 읽는다. 그래서 룰 로직을 mock으로
테스트하고, Redis 구현(`RedisUserBetHistory`)은 다음 커밋(006)에서 따로 붙인다 — 관심사 분리.
인터페이스를 좁게 둬서 각 룰이 필요한 것만 선언한다.

### RapidBettingRule — historicalCount + 1

```java
@Component
public class RapidBettingRule implements PatternRule {
  static final String NAME = "rapid-betting";
  private final RiskPatternProperties.RapidBetting cfg;
  private final UserBetHistory history;

  public RapidBettingRule(RiskPatternProperties patterns, UserBetHistory history) {
    this.cfg = patterns.rapidBetting();
    this.history = history;
  }

  @Override
  public Optional<PatternMatch> evaluate(PatternContext context) {
    if (!cfg.enabled()) return Optional.empty();
    long recent = history.countBetsBetween(
        context.userId(), context.now().minusSeconds(cfg.windowSeconds()), context.now());
    long withCandidate = recent + 1;                       // 후보 베팅 포함
    if (withCandidate < cfg.maxBets()) return Optional.empty();
    String reason = withCandidate + " bets in last " + cfg.windowSeconds() + "s reaches cap " + cfg.maxBets();
    return Optional.of(new PatternMatch(NAME, cfg.action(), reason));
  }
}
```

`historicalCount + 1`(후보 베팅 포함)을 maxBets와 비교. 그래서 `maxBets=30`이면 30번째
제출에서 발동 — 운영자 직관과 맞는다. disabled면 즉시 empty.

### SuddenStakeIncreaseRule — median 기반

```java
  @Override
  public Optional<PatternMatch> evaluate(PatternContext context) {
    if (!cfg.enabled()) return Optional.empty();
    List<Long> recent = history.recentStakeAmounts(context.userId(), cfg.lookbackBets());
    if (recent.size() < cfg.lookbackBets()) return Optional.empty();   // 신규 user: baseline 없음 → skip
    long median = median(recent);
    if (median <= 0) return Optional.empty();                          // zero-median: false fire 방지
    long candidate = context.stake().amount();
    if (candidate < (long) cfg.multiplierThreshold() * median) return Optional.empty();
    String reason = "stake " + candidate + " is " + cfg.multiplierThreshold()
        + "x or more above median " + median + " of last " + cfg.lookbackBets() + " bets";
    return Optional.of(new PatternMatch(NAME, cfg.action(), reason));
  }

  private static long median(List<Long> values) {
    List<Long> sorted = new ArrayList<>(values);
    Collections.sort(sorted);
    int size = sorted.size();
    if (size % 2 == 1) return sorted.get(size / 2);
    long lower = sorted.get(size / 2 - 1);
    long upper = sorted.get(size / 2);
    return lower + (upper - lower) / 2L;                               // 오버플로 피한 평균
  }
```

최근 stake의 **중앙값(median)**의 N배 이상이면 발동. 평균이 아니라 중앙값을 쓰는 건 정직한 큰
베팅 한 건이 baseline을 망치지 않게 하려고. lookback만큼 표본이 없으면(신규 user) 평가를
건너뛴다. 중앙값이 0이면(환불 폭탄 후 stake 이력이 전부 0) false fire를 막으려 short-circuit.
짝수 개수의 median은 `lower + (upper-lower)/2`로 구해 `(lower+upper)`의 오버플로를 피한다.

### RepeatedSameSelectionRule — 슬립 순회

```java
  @Override
  public Optional<PatternMatch> evaluate(PatternContext context) {
    if (!cfg.enabled()) return Optional.empty();
    Duration window = Duration.ofHours(cfg.windowHours());
    for (String selectionId : context.selectionIds()) {
      long prior = history.countSelectionBets(context.userId(), selectionId, window, context.now());
      long withCandidate = prior + 1;
      if (withCandidate > cfg.maxCount()) {
        String reason = "selection " + selectionId + " bet " + withCandidate
            + " times in last " + cfg.windowHours() + "h exceeds cap " + cfg.maxCount();
        return Optional.of(new PatternMatch(NAME, cfg.action(), reason));   // 첫 hot selection에서 멈춤
      }
    }
    return Optional.empty();
  }
```

슬립의 selectionId를 선언 순서로 훑다가 maxCount를 넘기는 **첫 selection에서 멈춘다**. reason
문자열이 그 selection을 콕 집어줘서 downstream이 룰을 재실행할 필요가 없다.

### 테스트 — Mockito strict stubbing 함정

`RepeatedSameSelectionRuleTest`는 룰이 selectionId 리스트를 훑으며 "hot"인 것만 stub하는데,
Mockito strict 기본값은 stub 안 한 "cold" 조회를 거부한다. 그 cold 조회는 기본 mock 답(0)으로
충분하므로 `@MockitoSettings(strictness = LENIENT)`로 완화했다. 17개 케이스(룰별 trigger/정상 +
엔진 fold/순서/BLOCK 비-short-circuit).
