# 011 · Redis 타임아웃 확대 (콜드스타트 connection init 대응)

> commit `b709e77` — `fix(config): widen risk redis timeouts to survive cold-start connection init`

## 개요

전 스택(docker-compose)을 **콜드(cold)로 동시 기동**한 통합 e2e에서 **첫 베팅이 HTTP 500**으로
실패했다. betting-service가 베팅 접수 직전에 부르는 `POST /internal/v1/risk/check`가
`RedisCommandTimeoutException`을 던졌고, betting은 그걸 read-timeout으로 받아 사용자에게 500을
돌려줬다. 이것이 시스템 통합에서만 드러난 9개 버그 중 **#9**다.

원인은 한 줄짜리 설정이었다. `spring.data.redis.timeout: 200ms`는 **command timeout**(명령 하나가
응답을 받기까지의 상한)인데, 이 값이 **콜드스타트의 connection init**(프로세스 부팅 직후 + JIT
워밍업 전 + 첫 GC + 초기 부하)을 견디지 못했다. 정상 상태의 check는 ~30 ms면 끝나지만, Lettuce가
**처음 한 번** TCP 연결을 수립하는 그 순간이 200 ms를 넘겼다. 즉 느린 건 명령 실행이 아니라
**최초 연결 수립**인데, 둘이 같은 200 ms 예산을 공유하고 있었던 게 문제다.

> 이건 제목이 "widen timeouts"로 넓어 보여도 실제 diff는 `application.yml` 한 곳, 6줄이다
> (자기 미화 금지). 코드 로직 변경은 없다.

## 작업 순서

1. 통합 e2e에서 첫 `/check` 500을 재현하고, risk 로그에서 `RedisCommandTimeoutException` 확인.
2. 느린 구간이 명령 실행이 아니라 **최초 connection 수립**임을 특정.
3. command timeout을 실패 상한으로 재정의(200ms → 1s)하고, connection 수립 예산을
   `connect-timeout: 5s`로 **분리**.
4. 정상 check latency(~30ms)가 불변임을 확인(상한만 넓힌 것이지 정상 경로는 그대로).

## 작업 내역

### 왜 단독 테스트는 이걸 못 잡았나 — 콜드 vs 워밍 기동

risk-service의 통합 테스트는 Testcontainers로 Redis를 띄운다. 그런데 Testcontainers는
컨테이너를 **재사용(reuse)**하고, 테스트가 첫 명령을 보낼 즈음이면 Redis는 이미 **워밍된(warm)**
상태다 — JVM도 테스트 러너가 이미 데우고 있고, 연결 풀(connection pool)도 사실상 준비돼 있다.
그래서 "프로세스가 막 떠서 처음으로 Redis에 연결하는 순간"이라는 타이밍 자체가 단독 테스트에는
**구조적으로 존재하지 않는다.** 이 버그는 전 스택을 콜드로 같이 올릴 때만 재현된다.

> C/C++ 관점 비유: 이건 "디버그 빌드에선 멀쩡한데 릴리스 빌드 첫 실행에서만 터지는" 부류, 혹은
> 콜드 캐시(cold cache) 첫 접근만 느린 상황과 같다. 단위 테스트 그린이 런타임 그린을 보장하지
> 못한다는 교훈은 008(commons-pool2 `NoClassDefFoundError`)에 이어 두 번째다 — 그때는 클래스패스,
> 이번엔 **기동 타이밍**이 단독 테스트의 사각이었다.

### Lettuce connection pool의 lazy 초기화

Spring Data Redis의 기본 클라이언트는 **Lettuce**다. C++에서 소켓을 명시적으로 `connect()`하는
것과 달리, Lettuce 연결은 **첫 명령이 올 때 게으르게(lazy) 수립**된다. 풀(`commons-pool2`,
008에서 추가)을 켜둬도 풀이 채워지는 시점은 부팅이 아니라 **처음 빌려갈 때**다. 그래서:

- 정상 운영 중: 연결은 이미 풀에 있고, 명령 하나의 왕복은 수 ms.
- 콜드 첫 요청: 풀이 비어 있어 **연결을 새로 맺어야** 하고, 그 한 번이 JVM 워밍업/GC와 겹치면
  수백 ms까지 튄다. 200 ms command timeout은 이 한 번을 명령 실패로 오판했다.

### command timeout vs connect timeout — 두 예산을 분리

핵심 수정은 **시간 예산(time budget)을 두 단계로 쪼갠 것**이다:

```yaml
# command timeout: failure ceiling, not the SLA. 200ms was too aggressive —
# cold-start connection init (boot, GC, load) exceeded it and produced false
# RedisCommandTimeoutException on the first /check (betting saw read-timeout 500s).
# Normal checks still return in ~30ms; this is just the upper bound.
timeout: 1s          # was 200ms — per-command response ceiling (failure bound)
connect-timeout: 5s  # new — socket/connection establishment budget, separate
```

- **`timeout`(command timeout)**: 이미 맺힌 연결 위에서 명령 하나가 응답을 받기까지의 상한.
  이건 **SLA가 아니라 "이만큼 넘으면 실패로 친다"는 상한선**이다. 정상 check가 1s를 쓰는 게
  아니라(여전히 ~30ms), 비정상 상황을 실패로 선언하기까지의 여유를 넓힌 것뿐이다.
- **`connect-timeout`**: 소켓/연결을 **수립**하는 단계의 별도 예산. 이게 따로 있어야 "연결 맺는 데
  오래 걸리는 것"과 "명령 실행이 오래 걸리는 것"을 더 이상 같은 200 ms로 뭉뚱그리지 않는다.

이 둘을 분리하는 게 수정의 본질이다. 숫자(1s/5s)는 **실패 상한**이지 목표 latency가 아니다.
정상 경로의 p99 < 30 ms 목표(009 부하 baseline)는 그대로 유효하고, 이 변경으로 바뀌지 않는다.

## 결과

콜드 기동 후 첫 `/check`가 더 이상 `RedisCommandTimeoutException`으로 죽지 않는다 — 최초 연결
수립이 1s command ceiling + 5s connect 예산 안에서 끝난다. 첫 베팅 500(통합 버그 #9)이 닫혔다.
정상 상태 latency는 불변(~30 ms).

## 요약

- 콜드스타트 connection init이 200ms command timeout을 초과 → 첫 `/check`가 false
  `RedisCommandTimeoutException` → betting read-timeout 500 (통합 버그 #9).
- command timeout 200ms→**1s**(실패 상한, SLA 아님), **`connect-timeout: 5s` 신설**로 연결 수립
  예산과 명령 실행 예산을 분리.
- 단독 테스트는 **워밍된** Testcontainers Redis라 콜드 첫 연결 타이밍을 구조적으로 미검출 —
  전 스택 콜드 기동에서만 재현.
- 정상 check latency(~30ms) 불변. diff는 `application.yml` 6줄, 코드 로직 변경 없음.

## 기억·설명 Level

- **L1**: command timeout 200ms→1s, connect-timeout 5s 신설. 콜드 첫 연결이 원인.
- **L2**: Lettuce 연결의 lazy 초기화 — 풀을 켜도 첫 요청에서야 연결이 맺힌다. 단독 테스트가
  못 잡은 이유(워밍된 Testcontainers Redis). 단위 그린 ≠ 런타임 그린(008에 이은 두 번째).
- **L3 (외워서)**: **command timeout vs connect timeout을 왜 분리했나** — 연결 수립 단계와
  명령 실행 단계는 서로 다른 시간 특성을 가지는데 하나의 200ms로 묶으면 콜드 connection init이
  정상 명령처럼 실패 판정된다. 1s/5s는 **실패 상한**이지 SLA가 아니라는 점, 정상 latency는
  불변이라는 점을 같이 말할 수 있어야 한다. 면접에서 "콜드스타트 이슈 겪은 적 있나?",
  "타임아웃을 어떻게 잡나?", "통합에서만 드러나는 버그 예시"로 직결.
