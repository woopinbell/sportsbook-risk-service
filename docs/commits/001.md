# 001 · 정책 바인딩 (RiskLimitProperties + RiskPatternProperties)

> commit `360ca13` — `feat(policy): bind risk limits and pattern rules from application.yml`

## 개요

application.yml의 `risk.limits` / `risk.patterns` 블록을 **type-safe하게 받는** 두 개의
`@ConfigurationProperties` record를 만든다. 이후 모든 계층(counter, resolver, rule, service)이
이 두 객체를 평범한 Spring 주입으로 받아 쓴다. ADR-0009의 "V1은 application.yml 기반 type-safe
config" 방향을 그대로 구현.

## 작업 순서

1. `RiskLimitProperties` record — 통화별 stake 한도 맵 + selections-per-minute.
2. `PatternAction` enum (SUSPECT / REVIEW / BLOCK) — 패턴 매치 결과 + 후속 단계 공용.
3. `RiskPatternProperties` record — 패턴 3종을 nested record로.
4. `ApplicationContextRunner` 기반 바인딩 테스트.

## 작업 내역

### RiskLimitProperties — 컴팩트 생성자에서 방어

```java
@ConfigurationProperties(prefix = "risk.limits")
public record RiskLimitProperties(
    Map<Currency, Long> stakeDaily,
    Map<Currency, Long> stakeWeekly,
    Map<Currency, Long> stakeMonthly,
    Map<Currency, Long> singleBetMax,
    Map<Currency, Long> openExposure,
    int selectionsPerMinute) {

  private static final long DEFAULT_KRW_DAILY = 1_000_000L;
  private static final long DEFAULT_USD_DAILY = 1_000L;
  // … weekly / monthly / single / exposure KRW·USD 기본 상수, DEFAULT_SELECTIONS_PER_MINUTE = 30 …

  public RiskLimitProperties {   // compact constructor
    stakeDaily = normalised(stakeDaily, DEFAULT_KRW_DAILY, DEFAULT_USD_DAILY);
    stakeWeekly = normalised(stakeWeekly, DEFAULT_KRW_WEEKLY, DEFAULT_USD_WEEKLY);
    stakeMonthly = normalised(stakeMonthly, DEFAULT_KRW_MONTHLY, DEFAULT_USD_MONTHLY);
    singleBetMax = normalised(singleBetMax, DEFAULT_KRW_SINGLE, DEFAULT_USD_SINGLE);
    openExposure = normalised(openExposure, DEFAULT_KRW_EXPOSURE, DEFAULT_USD_EXPOSURE);
    if (selectionsPerMinute <= 0) {
      selectionsPerMinute = DEFAULT_SELECTIONS_PER_MINUTE;
    }
  }

  public long stakeDaily(Currency currency) { return require(stakeDaily, currency, "stake-daily"); }
  // stakeWeekly / stakeMonthly / singleBetMax / openExposure 도 같은 require(...) 패턴

  private static Map<Currency, Long> normalised(
      Map<Currency, Long> source, long defaultKrw, long defaultUsd) {
    EnumMap<Currency, Long> copy = new EnumMap<>(Currency.class);
    if (source == null || source.isEmpty()) {
      copy.put(Currency.KRW, defaultKrw);
      copy.put(Currency.USD, defaultUsd);
      return Map.copyOf(copy);
    }
    source.forEach(
        (currency, amount) -> {
          if (amount == null || amount < 0) {
            throw new IllegalArgumentException(
                "Risk limit for " + currency + " must be non-negative, got " + amount);
          }
          copy.put(currency, amount);
        });
    copy.putIfAbsent(Currency.KRW, defaultKrw);
    copy.putIfAbsent(Currency.USD, defaultUsd);
    return Map.copyOf(copy);
  }

  private static long require(Map<Currency, Long> map, Currency currency, String limitName) {
    Long value = map.get(currency);
    if (value == null) {
      throw new IllegalStateException(
          "No " + limitName + " limit configured for currency " + currency);
    }
    return value;
  }
}
```

Java 17 record의 **컴팩트 생성자**는 C++ 생성자 초기화 리스트와 다르게, 필드 할당 *전에*
파라미터를 다듬는 자리다. 여기서 `normalised()`가 세 가지를 한다:

- yaml이 준 불변 맵을 `EnumMap` 기반 `Map.copyOf`로 **방어적 복사** — 외부에서 런타임 mutation 불가.
- 통화별 금액이 음수면 `IllegalArgumentException` → 부팅 거부.
- 특정 통화가 비어 있으면 `putIfAbsent`로 KRW/USD 기본값을 채움 → 운영자가 일부만 적어도 부팅 가능.

조회는 `stakeDaily(Currency)` 같은 메서드로 하고, 없으면 `require()`가 `IllegalStateException`.
**null을 조용히 반환하지 않는다** — C/C++에서 null 포인터를 늦게 터뜨리는 대신 호출 지점에서
즉시 명확한 예외를 던지는 쪽. 금액은 `long` minor unit(KRW 원, USD 센트)로 shared-protocol
`Money`와 일관.

ADR-0009는 override를 DB에 두라지만, 이 repo는 RDB가 없어 Redis hash로 둔다(004 커밋).
여기 yaml 값은 그 override가 fallback하는 기본값이다.

### PatternAction — 매치 결과 enum

```java
public enum PatternAction {
  SUSPECT,   // flag하되 베팅은 통과 — 분석가가 비동기 리뷰
  REVIEW,    // flag + 운영자 동기 리뷰 큐(admin-api UI)
  BLOCK      // 즉시 거절 — 한도 위반과 같은 critical-path 응답
}
```

top-level enum으로 빼서 properties·check API 응답(`patternsFlagged`)·`risk.pattern.suspected`
이벤트 라우팅이 **한 타입**을 공유한다. 패턴별 기본 action이 다르다는 점이 미묘하다 —
RepeatedSameSelection은 기본 `REVIEW`, 나머지 둘은 `SUSPECT`(아래).

### RiskPatternProperties — enabled일 때만 임계값 검증

```java
@ConfigurationProperties(prefix = "risk.patterns")
public record RiskPatternProperties(
    RapidBetting rapidBetting,
    SuddenStakeIncrease suddenStakeIncrease,
    RepeatedSameSelection repeatedSameSelection) {

  public RiskPatternProperties {
    if (rapidBetting == null) rapidBetting = RapidBetting.disabledDefault();
    if (suddenStakeIncrease == null) suddenStakeIncrease = SuddenStakeIncrease.disabledDefault();
    if (repeatedSameSelection == null) repeatedSameSelection = RepeatedSameSelection.disabledDefault();
  }

  public record RapidBetting(boolean enabled, int windowSeconds, int maxBets, PatternAction action) {
    public RapidBetting {
      if (action == null) action = PatternAction.SUSPECT;
      if (enabled) {   // 검증은 켜졌을 때만
        if (windowSeconds <= 0)
          throw new IllegalArgumentException("rapid-betting.window-seconds must be positive, got " + windowSeconds);
        if (maxBets <= 0)
          throw new IllegalArgumentException("rapid-betting.max-bets must be positive, got " + maxBets);
      }
    }
    static RapidBetting disabledDefault() {
      return new RapidBetting(false, DEFAULT_RAPID_WINDOW_SECONDS, DEFAULT_RAPID_MAX_BETS, PatternAction.SUSPECT);
    }
  }

  // SuddenStakeIncrease(enabled면 multiplierThreshold > 1, lookbackBets > 0; 기본 action SUSPECT)
  // RepeatedSameSelection(enabled면 windowHours > 0, maxCount > 0; 기본 action REVIEW)
  //   둘 다 같은 모양: action 기본값 + enabled-gated 검증 + disabledDefault() 팩토리.
}
```

검증을 `enabled`일 때만 거는 게 의도적이다. 운영자가 룰을 끄면서(`enabled: false`) 임계값을
0이나 placeholder로 남겨두는 게 runbook에서 흔한 패턴인데, 그걸 받아줘야 "한 줄 yaml 토글"이
어디서나 똑같이 동작한다. 켜져 있는데 임계값이 말이 안 되면 부팅을 거부한다. action 기본값은
보수적으로(SUSPECT/REVIEW, never BLOCK) — 문서 없는 yaml이 critical path에서 조용히 트래픽을
거절하기 시작하지 않게.

### 테스트 — ApplicationContextRunner

```java
new ApplicationContextRunner()
    .withUserConfiguration(EnableLimits.class)
    .withPropertyValues("risk.limits.stake-daily.KRW=2500000")
    .run(context -> assertThat(...));
```

`@SpringBootTest`로 전체 컨텍스트를 띄우는 대신 `ApplicationContextRunner`로 **properties
빈 하나만** 등록한 미니 컨텍스트를 돌린다. 약 50배 빠르고, `context.hasFailed()`로 바인딩
실패(음수 금액 등)도 검증할 수 있다. 10개 케이스: 기본값 fallback, override, 부분 통화 채움,
음수 거부, 비양수 selections fallback, 룰 3종 풀 바인딩, 전체 disabled 기본, enabled 임계값
거부 2건, disabled 룰의 placeholder 허용.

## 결과

`./mvnw clean verify` → 10/10 green. checkstyle MagicNumber가 disabledDefault 팩토리의
60/30/10/24/5를 잡아서 named constant로 빼고 통과.

## 요약

- record 컴팩트 생성자에서 방어적 복사 + 기본값 + 검증을 한 곳에.
- 조회 실패는 null이 아니라 예외.
- 패턴 임계값 검증은 `enabled`일 때만 — runbook 토글 친화.
- `ApplicationContextRunner`로 가볍고 빠른 바인딩 테스트.

## 기억·설명 Level

- **L1**: 어떤 정책 키가 있는지(stake 일/주/월, single-bet, open-exposure, selections, 패턴 3종).
- **L2**: 컴팩트 생성자에서 방어적 복사·기본값 처리, `ApplicationContextRunner`가 `@SpringBootTest`보다
  싼 이유.
- **L3**: "패턴 검증을 왜 enabled일 때만?"(runbook 토글), record 기반 `@ConfigurationProperties`가
  `@ConstructorBinding` 없이 동작하는 Spring Boot 3 바인딩. 면접에서 "설정을 어떻게 안전하게
  받나"로 직결.
