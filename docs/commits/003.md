# 003 · Redis sliding window counter (sorted set + sum key + Lua)

> commit `03ee3c8` — `feat(counter): atomic sliding-window counter on Redis sorted set + sum key`

## 개요

이 repo의 핵심 저장 원시 타입(storage primitive). 시간 윈도우 한도(일/주/월 stake, 분당
selection 수)가 전부 이 `SlidingWindowCounter` 위에 선다. 자료구조는 **Redis sorted set +
별도 sum key 쌍**, 만료 정리와 합산은 **Lua script로 atomic**하게.

왜 sliding window인가: tumbling window(자정마다 리셋)는 자정 직후 같은 user가 다시 한도만큼
베팅하는 게이밍이 가능하다. sliding은 "정확히 지난 24시간"이라 사용자 친화적이고 우회가 어렵다.

왜 Redis인가: Kafka KTable 같은 stateful streaming은 학습 가치는 있지만 V1엔 오버킬(ADR 단계
판단). sorted set이 빠르고 단순하며 p99 < 30 ms 목표에 맞는다.

## 작업 순서

1. `LimitType` enum — 한도 종류 ↔ 키 suffix ↔ 윈도우 길이 ↔ Measure(SUM/COUNT).
2. `LimitKeys` — Redis 키 컨벤션 단일 출처.
3. `scripts/sliding-window.lua` — 정리 + 기록 + 합산을 한 번에.
4. `SlidingWindowCounter` — 스크립트를 감싸는 Spring 컴포넌트.
5. Testcontainers Redis 통합 테스트(동시성 race 포함).

## 작업 내역

### 자료구조

```
limit:user:{userId}:{suffix}        sorted set
    score  = 베팅 시각 (epoch ms)
    member = "{betId}|{amount}"      amount는 long minor unit
limit:user:{userId}:{suffix}:sum    long counter, sorted set과 lockstep 유지
```

sorted set만으로도 윈도우 내 베팅을 `ZRANGEBYSCORE`로 셀 수 있지만, **합산하려면 모든 멤버를
훑어야** 한다. check API는 critical path라 O(N) 스캔이 부담이다. 그래서 sum key를 둬서
O(1) 조회를 만든다. 문제는 "sum key가 sorted set과 어긋나지 않게 유지"하는 것 — 여기서 Lua가 핵심.

### LimitType — 한도 메타데이터

```java
public enum LimitType {
  STAKE_DAILY("stake-daily", Duration.ofDays(1), Measure.SUM),
  STAKE_WEEKLY("stake-weekly", Duration.ofDays(7), Measure.SUM),
  STAKE_MONTHLY("stake-monthly", Duration.ofDays(30), Measure.SUM),
  SELECTIONS_PER_MINUTE("selections-per-minute", Duration.ofMinutes(1), Measure.COUNT);

  private final String suffix;
  private final Duration window;
  private final Measure measure;

  LimitType(String suffix, Duration window, Measure measure) {
    this.suffix = suffix;
    this.window = window;
    this.measure = measure;
  }

  public String suffix() { return suffix; }
  public Duration window() { return window; }
  public Measure measure() { return measure; }

  public enum Measure { SUM, COUNT }
}
```

각 enum 상수가 (키 suffix, 윈도우 길이, 측정 방식)을 한데 묶는다. `suffix()`가 Redis 키의
꼬리를, `window()`가 sliding 길이를 준다. `SINGLE_BET_MAX`와 `OPEN_EXPOSURE`는 **일부러
뺐다**. single-bet-max는 윈도우 누적이 아니라 베팅당 임계값이고(service에서 직접 비교),
open-exposure는 증감하는 live state라 sorted set TTL 모델과 안 맞는다. `Measure`는 정보성
힌트다 — SUM은 member에 amount를 인코딩하고, COUNT는 같은 스크립트를 amount=1로 써서 **하나의
Lua script**만 유지한다.

### LimitKeys — 키 컨벤션 단일 출처

```java
public final class LimitKeys {

  private static final String USER_PREFIX = "limit:user:";
  private static final String MARKET_PREFIX = "limit:market:";
  private static final String SUM_SUFFIX = ":sum";

  private LimitKeys() {}

  public static String userKey(String userId, LimitType type) {
    return USER_PREFIX + userId + ":" + type.suffix();
  }

  public static String marketKey(String marketId, LimitType type) {
    return MARKET_PREFIX + marketId + ":" + type.suffix();
  }

  public static String sumKey(String slidingKey) {
    return slidingKey + SUM_SUFFIX;
  }

  public static List<String> userKeyPair(String userId, LimitType type) {
    String slidingKey = userKey(userId, type);
    return List.of(slidingKey, sumKey(slidingKey));
  }

  /** Encodes the per-member token the Lua script decodes back when an entry expires. */
  public static String encodeMember(String betId, long amount) {
    return betId + "|" + amount;
  }
}
```

키 컨벤션을 한 곳에 모은 이유: counter·check API·override store가 전부 같은 키 모양을 빌드해야
하므로, 컨벤션이 바뀌어도 한 파일만 고치면 된다. `sumKey`는 sliding key에 `:sum`을 붙이는
**한 규칙**이라 sorted set과 sum key가 항상 짝으로 움직인다. `encodeMember`가 만든
`"{betId}|{amount}"` 토큰을 Lua가 만료 시 파싱해 amount를 되돌린다 — 인코딩/디코딩 규칙이
두 언어(Java/Lua)에 걸쳐 있으므로 이 한 줄이 그 계약이다.

### Lua script — 정리·기록·합산을 한 atomic으로

C/C++로 치면 여러 Redis 명령을 하나의 critical section으로 묶는 일이다. Redis는 단일 스레드라
Lua script 실행 중에는 다른 명령이 끼어들지 못한다(atomic). 스크립트는 한 번의 호출에서
만료 멤버를 찾아 sum에서 빼고(DECRBY) sorted set에서 지우고(ZREM), 기록 호출이면 새 멤버를
ZADD + sum INCRBY 한 뒤, 두 key의 TTL을 갱신하고 현재 sum을 반환한다.

```lua
-- KEYS[1] = sorted set key,  KEYS[2] = sum key
-- ARGV = now(ms), window(ms), member("<betId>|<amount>" 또는 ""), amount(peek=0), ttlSeconds
local nowMs = tonumber(ARGV[1])
local windowMs = tonumber(ARGV[2])
local member = ARGV[3]
local amount = tonumber(ARGV[4])
local ttlSeconds = tonumber(ARGV[5])

local cutoff = nowMs - windowMs

-- 1. 윈도우를 벗어난 멤버를 찾아 인코딩된 amount를 합산해 sum에서 DECRBY, 그리고 ZREM.
--    per-member 회계를 먼저 끝내야 만료가 한꺼번에 몰려도 sum이 sorted set과 lockstep을 유지한다.
local expired = redis.call('ZRANGEBYSCORE', KEYS[1], '-inf', '(' .. tostring(cutoff))
if #expired > 0 then
    local expiredSum = 0
    for i = 1, #expired do
        local m = expired[i]
        local pipe = string.find(m, '|', 1, true)
        if pipe ~= nil then
            local encoded = tonumber(string.sub(m, pipe + 1))
            if encoded ~= nil then
                expiredSum = expiredSum + encoded
            end
        end
    end
    if expiredSum > 0 then
        redis.call('DECRBY', KEYS[2], expiredSum)
    end
    redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', '(' .. tostring(cutoff))
end

-- 2. record 호출이면 새 멤버 추가(peek은 빈 member + amount=0이라 건너뛴다).
if amount > 0 and member ~= '' then
    redis.call('ZADD', KEYS[1], nowMs, member)
    redis.call('INCRBY', KEYS[2], amount)
end

-- 3. 두 key의 TTL 갱신(없는 key엔 no-op이라 무조건 호출해도 안전).
redis.call('EXPIRE', KEYS[1], ttlSeconds)
redis.call('EXPIRE', KEYS[2], ttlSeconds)

-- 4. 현재 sum 반환(sum key가 없으면 0).
local current = redis.call('GET', KEYS[2])
if current == false then
    return 0
end
return tonumber(current)
```

**불변식(invariant)**: sum key == sorted set에 살아있는 멤버들의 amount 합. 모든 쓰기가 이
스크립트를 통과하는 한, 만료 정리의 DECRBY와 ZREM이 한 atomic 단계에 묶이므로 reader는
sum이 sorted set보다 앞서거나 뒤처진 상태를 절대 못 본다.

read-only peek은 빈 member + amount=0으로 모델링했다. 그래서 read와 write가 **같은 스크립트**를
타고, 둘 다 같은 정리·같은 EXPIRE 갱신·같은 사후 sum 반환을 한다. 스크립트가 하나라 추론이 쉽다.

TTL은 `max(60s, 2 * window)`. 조용하던 user가 윈도우 안에 다시 돌아왔을 때, Redis가 cold key를
이미 만료시켜 상태를 잃는 일을 막는다.

### SlidingWindowCounter — 스크립트를 감싸는 컴포넌트

```java
@Component
public class SlidingWindowCounter {

  private static final Duration TTL_SAFETY_MULTIPLIER = Duration.ZERO;
  private static final long TTL_MULTIPLIER = 2L;
  private static final long MIN_TTL_SECONDS = 60L;
  private static final String READ_ONLY_MEMBER = "";
  private static final long READ_ONLY_AMOUNT = 0L;

  private final StringRedisTemplate redis;
  private final Clock clock;
  private final RedisScript<Long> script;

  public SlidingWindowCounter(StringRedisTemplate redis, Clock clock) {
    this.redis = redis;
    this.clock = clock;
    this.script = new DefaultRedisScript<>(loadScript(), Long.class);
  }

  public long record(
      String slidingKey, String memberToken, long amount, Duration window, Instant now) {
    if (amount <= 0) {
      throw new IllegalArgumentException("amount must be positive for record(), got " + amount);
    }
    return execute(slidingKey, memberToken, amount, window, now);
  }

  /** Returns the current windowed sum without recording — still runs cleanup for read/write parity. */
  public long currentSum(String slidingKey, Duration window, Instant now) {
    return execute(slidingKey, READ_ONLY_MEMBER, READ_ONLY_AMOUNT, window, now);
  }

  private long execute(
      String slidingKey, String memberToken, long amount, Duration window, Instant now) {
    List<String> keys = List.of(slidingKey, LimitKeys.sumKey(slidingKey));
    long ttlSeconds = ttlSecondsFor(window);
    Long result =
        redis.execute(
            script, keys,
            Long.toString(now.toEpochMilli()),
            Long.toString(window.toMillis()),
            memberToken,
            Long.toString(amount),
            Long.toString(ttlSeconds));
    return result == null ? 0L : result;
  }

  private static long ttlSecondsFor(Duration window) {
    return Math.max(
        MIN_TTL_SECONDS, window.plus(TTL_SAFETY_MULTIPLIER).getSeconds() * TTL_MULTIPLIER);
  }

  private static String loadScript() {
    try {
      return StreamUtils.copyToString(
          new ClassPathResource("scripts/sliding-window.lua").getInputStream(),
          StandardCharsets.UTF_8);
    } catch (Exception e) {
      throw new IllegalStateException("Failed to load sliding-window.lua from classpath", e);
    }
  }
}
```

`record`와 `currentSum`이 **같은 `execute`**로 내려가고, peek은 빈 member/amount=0 상수를
넘겨 Lua의 read 경로를 탄다. `DefaultRedisScript<>(…, Long.class)`로 묶어 Spring이 EVALSHA
캐싱을 처리하고, 스크립트 본문은 classpath의 `.lua`에서 한 번 읽어 둔다(인라인 문자열이 아니라
파일이라 diff·문법 하이라이트가 쉽다). `record`만 amount>0을 강제하고 — peek은 0이 정상이라
가드가 `record`에만 있다. 각 메서드는 `Clock` 주입 overload도 둬서 테스트가 fixed clock으로
윈도우 경계를 강제할 수 있다.

### 테스트 — 동시성 race가 테스트 자신을 deadlock시킨 사건

```java
// 100개 동시 record가 정확히 100 * stake로 합산되는지
ExecutorService pool = Executors.newFixedThreadPool(threads);  // ← threads=100
CountDownLatch ready = ..., start = ..., done = ...;
```

처음엔 `newFixedThreadPool(16)`에 100개 task를 submit했다. 그런데 worker 16개가 전부
`start.await()`에 묶이면 나머지 84개는 pool에 들어가지도 못해 `ready` latch가 100에 도달하지
못한다 — **테스트가 매달렸다(hang)**. surefire가 안 끝나서 `jstack`으로 main 스레드가
`LockSupport.park`(ready.await)에 묶인 걸 보고 진단했다. 고친 건 pool 크기를 task 수와 같게
한 줄. 교훈: latch 안무를 쓰는 동시성 테스트는 pool이 task 수만큼 커야 starvation이 없다.

6개 케이스: record→peek 일치, 윈도우 내 2건 합산, 시계가 윈도우 경계를 넘으며 만료, COUNT
스타일(amount=1), 음수/0 거부, **100 동시 record → 정확한 합**(이게 Level 3 정합성 증명).

## 요약

- sorted set(score=시각) + sum key(O(1) 조회) 쌍.
- Lua로 정리+기록+합산을 atomic하게 → sum 불변식 유지.
- read/write가 같은 스크립트(peek = 빈 member).
- single-bet-max/open-exposure는 윈도우 모델이 아니라 enum에서 제외.
- 동시성 테스트 deadlock → pool 크기 = task 수.

## 기억·설명 Level

- **L1**: 키 두 개(sorted set + :sum)가 있다는 사실, TTL = 2 * window.
- **L2**: peek이 빈 member로 같은 스크립트를 타는 설계, COUNT/SUM이 한 스크립트인 점,
  동시성 테스트 deadlock의 원인과 수정.
- **L3 (외워서)**: **sum key 불변식과 Lua atomic의 관계** — "왜 sum이 sorted set과 안
  어긋나는가"를 DECRBY+ZREM 원자성으로 설명. sliding vs tumbling 선택 이유. read-only check의
  TOCTOU 틈(두 동시 베팅이 한도 직전 상태를 같이 읽으면 둘 다 통과 가능 — V1이 용인하는 한계).
  면접 단골: "분산 환경에서 카운터 정합성을 어떻게 보장?"
