# 005 · 패턴 룰 엔진 (전략 패턴)

## 개요

한도 검사와 나란히 가는 **의심 패턴 탐지** 계층. V1 룰 3종을 전략 패턴(strategy pattern)으로
구현하고, `RuleEngine`이 이들을 fold한다. ML은 범위 밖(CLAUDE.md) — 룰 기반만.

## 작업 순서

1. `PatternContext`(입력) / `PatternMatch`(판정) / `PatternRule`(전략 인터페이스) 정의.
2. `UserBetHistory` — 룰이 읽는 좁은 read 인터페이스(이 커밋엔 구현 없음, mock으로 테스트).
3. 룰 3종 — `RapidBettingRule` / `SuddenStakeIncreaseRule` / `RepeatedSameSelectionRule`.
4. `RuleEngine` — `List<PatternRule>` 주입받아 fold.
5. Mockito 단위 테스트.

## 작업 내역

### 계약 — PatternContext / PatternMatch / PatternRule

**책임:** 룰이 베팅 하나를 평가할 때 보는 immutable 스냅샷(`now`를 명시 파라미터로 받아 fixed
clock 테스트 가능), 룰의 판정(yaml 키와 같은 ruleName + action + locale-free reason), 전략
인터페이스(미발동 = empty)를 정의한다. 셋 다 계약이므로 헤더는 노출, 방어 로직만 비움.

```java
public record PatternContext(
    String userId, String betId, Money stake, List<String> selectionIds, Instant now) {
  public PatternContext {
    // TODO: selectionIds 불변 방어 복사 (룰이 ctx를 mutate 못 하게)
  }
}

public record PatternMatch(String ruleName, PatternAction action, String reason) {}

public interface PatternRule {
  Optional<PatternMatch> evaluate(PatternContext context);   // 미발동 = Optional.empty()
}
```

### RuleEngine — 전략 패턴 + Spring 자동 수집

### UserBetHistory — 좁은 read 인터페이스

**책임:** 룰이 읽는 read-only 뷰. 좁게 둬서 각 룰이 필요한 것만 선언하고, Redis 구현(006)을
인메모리 더블로 교체 가능하게 한다. (순수 인터페이스 = 계약이므로 그대로 노출.)

```java
public interface UserBetHistory {
  long countBetsBetween(String userId, Instant from, Instant to);                    // rapid-betting
  List<Long> recentStakeAmounts(String userId, int lookback);                        // sudden-stake (oldest-first)
  long countSelectionBets(String userId, String selectionId, Duration window, Instant now); // repeated
}
```

30번째에서). disabled면 즉시 empty. NAME·cfg·history 주입은 전략 패턴 wiring(노출), 판정
