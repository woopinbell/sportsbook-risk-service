# 009 · 부하/증명 baseline (check API + Kafka produce)

## 개요

`load-test/` 트리를 추가하고 baseline을 측정해 박제한다. check API latency는 k6,
`bet.placed` produce throughput은 Confluent의 `kafka-producer-perf-test`. 결과는 날짜 폴더 +
`BEST.md`에.

## 작업 순서

1. `docker-compose.yml` — Redis(6390) + 단일 노드 Kafka(KRaft).
2. `scenarios/check_latency.js` — k6 2단(realistic + saturation).
3. `scenarios/consumer_throughput.sh` — perf-test 래퍼.
4. 환경 부팅 → risk-service jar 띄우기 → 시나리오 실행 → 결과 박제.

## 작업 내역

### docker-compose — Redis + KRaft Kafka (두 listener)

**책임:** 부하 전용 Redis + 단일 노드 KRaft Kafka. dev와 충돌 안 나게 포트 bump. **두 함정**:
(1) cluster id는 유효한 base64 22자 UUID여야 broker가 뜬다. (2) **localhost 함정** — 컨테이너
*안*의 perf-test와 *밖*의 risk-service가 같은 broker에 닿으려면 advertised listener를
INTERNAL(컨테이너 hostname)/HOST(localhost)로 분리해야 한다. 서비스 키·구조는 노출, 핵심
값(포트·listener 주소·cluster id)은 비움.

```yaml
services:
  redis:
    image: redis:7-alpine
    ports: ["___:6379"]          # TODO: dev 기본 6379와 충돌 안 나는 host 포트
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
  kafka:
    image: confluentinc/cp-kafka:7.6.1
    container_name: risk-load-kafka
    ports: ["___:9092"]          # TODO: host 노출 포트
    environment:
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      # TODO: KAFKA_LISTENERS — INTERNAL/HOST/CONTROLLER 세 listener (0.0.0.0 바인드)
      # TODO: KAFKA_ADVERTISED_LISTENERS — INTERNAL://{컨테이너hostname}:9092, HOST://localhost:{port}
      # TODO: KAFKA_LISTENER_SECURITY_PROTOCOL_MAP / CONTROLLER_LISTENER_NAMES /
      #       CONTROLLER_QUORUM_VOTERS / INTER_BROKER_LISTENER_NAME
      # TODO: CLUSTER_ID — 유효한 base64 22자 UUID (아니면 broker 부팅 실패)
```

### check_latency.js — realistic vs saturation 2단

**책임:** 한 번의 k6 실행으로 두 숫자를 잡는다. **realistic_1000rps**(constant-arrival-rate)는
k6가 VU를 도착률에 맞춰 조절 → latency가 서비스 작업을 반영(p99<30ms 목표). **saturation_5000vu**
(constant-vus)는 queue cliff만 보여줌(SLO 없음). userId를 흩어 hot key를 피하는 게 핵심.
시나리오/threshold **키 구조**는 노출, 값과 요청 본문은 비움.

### consumer_throughput.sh — perf-test 래퍼

**책임:** k6엔 Kafka producer가 없어 cp-kafka 컨테이너의 `kafka-producer-perf-test`로 produce
throughput만 잰다(합성 바이트라 consumer는 Avro decode 실패 — 의도됨). 컨테이너 내부 exec라
INTERNAL listener를 쓴다. env 기본값·구조는 노출, perf-test 인자는 비움.

```bash
set -euo pipefail
RECORDS=${RECORDS:-10000}
RATE=${RATE:--1}
PAYLOAD_SIZE=${PAYLOAD_SIZE:-256}
TOPIC=${TOPIC:-bet.placed}
INTERNAL_BOOTSTRAP=${INTERNAL_BOOTSTRAP:-risk-load-kafka:9092}   # 컨테이너 내부 → INTERNAL listener
```

## 검증

- 환경 부팅: `docker compose -f load-test/docker-compose.yml up -d` → Redis/Kafka healthcheck green.
  (cluster id가 틀리면 kafka 컨테이너가 부팅 실패하는지 확인 — 함정 1.)
- risk-service jar를 host에서 띄우고(8083, REDIS_PORT=6390, KAFKA_BOOTSTRAP=localhost:9094)
  `k6 run scenarios/check_latency.js` → realistic threshold가 평가되는가(p99 라인).
- `./scenarios/consumer_throughput.sh` → 컨테이너 내부 exec가 INTERNAL listener로 broker에
  닿아 produce throughput을 내는가(localhost 함정 회피 확인).
