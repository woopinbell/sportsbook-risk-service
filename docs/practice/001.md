# 001 · 정책 바인딩 (RiskLimitProperties + RiskPatternProperties)

## 개요

application.yml의 `risk.limits` / `risk.patterns` 블록을 **type-safe하게 받는** 두 개의
`@ConfigurationProperties` record를 만든다. 이후 모든 계층(counter, resolver, rule, service)이
이 두 객체를 평범한 Spring 주입으로 받아 쓴다. ADR-0009의 "V1은 application.yml 기반 type-safe
config" 방향을 그대로 구현.

## 작업 순서

1. `RiskLimitProperties` record — 통화별 stake 한도 맵 + selections-per-minute.
2. `PatternAction` enum (SUSPECT / REVIEW / BLOCK) — 패턴 매치 결과 + 후속 단계 공용.
3. `RiskPatternProperties` record — 패턴 3종을 nested record로.
4. `ApplicationContextRunner` 기반 바인딩 테스트.

## 작업 내역

### RiskLimitProperties — 컴팩트 생성자에서 방어

**책임:** yaml `risk.limits`를 받아 통화별 한도 맵을 **방어적 복사 + 빠진 통화 기본값 + 음수
거부**로 다듬는다(컴팩트 생성자). 조회는 null이 아니라 예외로 실패한다(fail loud). 컴포넌트
목록(record 헤더)은 바인딩 계약이므로 노출, 다듬는 로직은 비움.

### PatternAction — 매치 결과 enum

**책임:** 패턴 매치의 후속 처리 단계를 표현하는 enum. properties·check 응답·이벤트 라우팅이
공유하는 계약이다. 세 단계의 의미(통과/리뷰/거절)를 구분하라.

```java
public enum PatternAction {
  // TODO: SUSPECT(flag하되 베팅 통과) / REVIEW(운영자 동기 리뷰 큐) / BLOCK(즉시 거절, critical-path)
}
```

### RiskPatternProperties — enabled일 때만 임계값 검증

**책임:** 패턴 3종을 nested record로 받는다. 각 룰은 독립 토글 + 자체 임계값. 핵심 결정:
**검증은 `enabled`일 때만** 건다(runbook이 끈 룰에 placeholder 임계값을 남기는 걸 허용). action
기본값은 보수적(SUSPECT/REVIEW, never BLOCK). 누락 nested는 disabledDefault()로. record 헤더
(컴포넌트 목록)는 계약이므로 노출.

```java
@ConfigurationProperties(prefix = "risk.patterns")
public record RiskPatternProperties(
    RapidBetting rapidBetting,
    SuddenStakeIncrease suddenStakeIncrease,
    RepeatedSameSelection repeatedSameSelection) {

  public RiskPatternProperties {
    // TODO: 각 nested가 null이면 해당 disabledDefault()로 대체
  }

  public record RapidBetting(boolean enabled, int windowSeconds, int maxBets, PatternAction action) {
    public RapidBetting {
      // TODO: action null이면 SUSPECT. enabled일 때만 windowSeconds>0, maxBets>0 검증(아니면 IllegalArgumentException).
    }
    static RapidBetting disabledDefault() {
      // TODO: enabled=false + 기본 window/maxBets + SUSPECT
    }
  }

  // SuddenStakeIncrease(boolean enabled, int multiplierThreshold, int lookbackBets, PatternAction action)
  //   → enabled면 multiplierThreshold > 1, lookbackBets > 0 검증. 기본 action SUSPECT.
  // RepeatedSameSelection(boolean enabled, int windowHours, int maxCount, PatternAction action)
  //   → enabled면 windowHours > 0, maxCount > 0 검증. 기본 action REVIEW.  ← action 기본값이 다름에 주의.
  //   두 record 모두 RapidBetting과 같은 모양(action 기본값 + enabled-gated 검증 + disabledDefault())으로 작성.
}
```

### 테스트 — ApplicationContextRunner

**책임:** `@SpringBootTest` 대신 `ApplicationContextRunner`로 properties 빈 하나만 띄운 미니
컨텍스트에서 바인딩을 검증한다(약 50배 빠름, `context.hasFailed()`로 부팅 실패도 검사). 10개
케이스: 기본값 fallback, override, 부분 통화 채움, 음수 거부, 비양수 selections fallback, 룰
3종 풀 바인딩, 전체 disabled 기본, enabled 임계값 거부 2건, disabled 룰의 placeholder 허용.

## 검증

- `./mvnw test -Dtest=RiskLimitPropertiesTest,RiskPatternPropertiesTest` → 10/10 green.
- 감각 점검: yaml에 `stake-daily.KRW`만 주면 USD는 기본값으로 채워지는가. 음수 금액이면 부팅이
  거부되는가(`getStartupFailure().rootCause()`가 IllegalArgumentException). `enabled: false`인
  룰에 `max-bets: 0`을 둬도 바인딩이 통과하는가(검증은 enabled일 때만).
