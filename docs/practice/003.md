# 003 · Redis sliding window counter (sorted set + sum key + Lua)

## 개요

이 repo의 핵심 저장 원시 타입(storage primitive). 시간 윈도우 한도(일/주/월 stake, 분당
selection 수)가 전부 이 `SlidingWindowCounter` 위에 선다. 자료구조는 **Redis sorted set +
별도 sum key 쌍**, 만료 정리와 합산은 **Lua script로 atomic**하게.

왜 sliding window인가: tumbling window(자정마다 리셋)는 자정 직후 같은 user가 다시 한도만큼
베팅하는 게이밍이 가능하다. sliding은 "정확히 지난 24시간"이라 사용자 친화적이고 우회가 어렵다.

왜 Redis인가: Kafka KTable 같은 stateful streaming은 학습 가치는 있지만 V1엔 오버킬(ADR 단계
판단). sorted set이 빠르고 단순하며 p99 < 30 ms 목표에 맞는다.

## 작업 순서

1. `LimitType` enum — 한도 종류 ↔ 키 suffix ↔ 윈도우 길이 ↔ Measure(SUM/COUNT).
2. `LimitKeys` — Redis 키 컨벤션 단일 출처.
3. `scripts/sliding-window.lua` — 정리 + 기록 + 합산을 한 번에.
4. `SlidingWindowCounter` — 스크립트를 감싸는 Spring 컴포넌트.
5. Testcontainers Redis 통합 테스트(동시성 race 포함).

## 작업 내역

### 자료구조

**책임:** 윈도우 내 합산을 O(1)로 만들기 위한 키 컨벤션을 정한다 — sorted set 하나로는 합산이
O(N)이라, 짝이 되는 sum key를 둔다. 이 두 key가 어긋나지 않게 유지하는 게 이후 Lua의 과제다.
(아래는 그 계약. 구현 대상이 아니라 시작점이다.)

```
limit:user:{userId}:{suffix}        sorted set
    score  = 베팅 시각 (epoch ms)
    member = "{betId}|{amount}"      amount는 long minor unit
limit:user:{userId}:{suffix}:sum    long counter, sorted set과 lockstep 유지
```

### Lua script — 정리·기록·합산을 한 atomic으로

**책임:** Redis 단일 스레드 = 스크립트 실행 중 끼어듦 없음(atomic)을 이용해, 만료 정리·기록·
합산을 한 호출에 묶는다. read(peek)와 write(record)가 **같은 스크립트**를 타도록 만들어 둘이
같은 정리·같은 TTL 갱신을 거치게 한다. 불변식 — sum key == 살아있는 멤버 amount 합 — 을 깨지
않게 만료 회계(DECRBY)와 ZREM을 한 단계에 둔다. (KEYS/ARGV 계약은 노출, 4단계 본문은 비움.)

```lua

-- Returns: 정리/기록 후의 현재 윈도우 sum
local nowMs = tonumber(ARGV[1])
local windowMs = tonumber(ARGV[2])
local member = ARGV[3]
local amount = tonumber(ARGV[4])
local ttlSeconds = tonumber(ARGV[5])

local cutoff = nowMs - windowMs

--         합산 → KEYS[2] DECRBY → KEYS[1] ZREMRANGEBYSCORE. (per-member 회계를 먼저 끝내 lockstep 유지)

-- TODO 2: record 호출이면(amount>0 and member~='') KEYS[1] ZADD(score=nowMs) + KEYS[2] INCRBY.
```

### 테스트 — 동시성 race가 테스트 자신을 deadlock시킨 사건

**책임:** Testcontainers Redis로 6개 케이스를 건다 — record→peek 일치, 윈도우 내 2건 합산,
fixed clock으로 윈도우 경계 만료, COUNT 스타일(amount=1), 음수/0 거부, **100 동시 record →
정확한 합**(정합성 증명). 마지막 케이스의 함정: latch 안무(ready/start/done)를 쓰는 동시성
테스트는 thread pool 크기가 task 수보다 작으면 worker가 전부 `start.await()`에 묶여 `ready`가
도달하지 못하고 **테스트가 hang**한다. pool 크기 = task 수로 맞춰라.

## 검증

- `./mvnw test -Dtest=SlidingWindowCounterTest` (Docker 필요 — Testcontainers Redis) → 6/6 green.
- 불변식 감각 점검: `record` 두 번 → `currentSum`이 두 amount의 합인가. 윈도우를 넘긴 fixed
  clock에서 만료분이 sum에서 빠지는가. peek은 멤버를 추가하지 않는가(빈 member 경로).
- 동시성: 100 동시 record의 최종 sum이 정확히 `100 * stake`인가(double-count/lost-update 0).
