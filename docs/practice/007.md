# 007 · Kafka event sourcing + 위험 이벤트 publish

## 개요

서비스의 **write side**. betting-service가 발행하는 `bet.placed`를 consume해 sliding window
counter와 패턴 history를 채우고(event sourcing), check에서 나온 위반/패턴을
`risk.limit.violated` / `risk.pattern.suspected`로 publish한다. 이걸로 push 모델(HTTP pull이
아니라 Kafka)이 완성된다 — betting-service에 부담을 안 주고 자체 집계를 유지.

## 작업 순서

1. `AvroCodec` — SpecificRecord ↔ binary 단일 코덱.
2. `EventTopics` — 토픽 이름 `@ConfigurationProperties`.
3. `BetPlacedConsumer` — `@KafkaListener`, counter/history 갱신.
4. `RiskEventPublisher` — 내부 모델 → wire 스키마 번역 후 publish.
5. `RiskCheckService`에 publisher 주입.
6. Mockito 단위 테스트 + Avro 왕복 테스트.

### Avro generated 타입이 가정과 달랐다 (재작업)

**책임:** publisher/consumer를 구현하다 만나는 함정. shared-protocol Avro plugin이
`timestamp-millis`를 `long`이 아니라 **`java.time.Instant`**로 매핑(jsr310)한다.
`setOccurredAt(long)` / `getRequestedAt()`가 long이라 가정하면 컴파일 에러. sources jar를 풀어
generated setter 시그니처를 직접 확인하고 `Instant`로 맞춰라 — 의존 라이브러리 generated 코드는
가정하지 말 것.
